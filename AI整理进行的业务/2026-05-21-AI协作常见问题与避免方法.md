# AI 协作开发常见问题与避免方法

> 整理自：10647 货币盈利及时反馈优化（建筑银元飘字）开发过程
> 日期：2026-05-21

---

## 一、需求阶段

### 问题 1：需求中途变更导致多次返工

**案例：** 飘字触发时机最初定义为"服务端推送恢复完成消息时触发"，代码按照这个方案完成（缓存价格 → Done 处理器发送事件）。之后需求调整为"英雄付费时触发"，导致三个文件需要重写：移除缓存字典、移动事件发送位置、删除 Done 处理器中的事件代码。

**问题根因：** 需求澄清阶段没有把关键时序问题讲透。"什么时候触发"是最核心的交互问题，应该在需求文档定稿前彻底确认。

**避免方法：**
- 需求澄清阶段，对涉及"时机""触发条件"的决策，显式列出所有候选时机并逐一讨论
- 在需求文档中明确标注时序决策点，要求业务方确认后再进入编码
- 检查清单：在开工前问自己——「这个动作是在 A 时刻做还是 B 时刻做？确认了吗？」

## 二、编码阶段

### 问题 2：Subagent 对项目技术栈理解不足（基类错误）

**案例：** Subagent 生成 `HeroEquipmentTradeDialogItemController` 和 `BuildingSilverFloatManager` 时，使用了标准 `MonoBehaviour` 作为基类。但项目使用 `IBaseController`（继承 MonoBehaviour 并实现 IController），它提供了 `SendEvent` / `RegisterEvent` 等事件接口。基类错误导致代码无法编译。

**问题根因：** Subagent 的 prompt 中没有明确指定项目的基类规范。Subagent 用 Unity 标准做法替代了项目特定做法。

**避免方法：**
- 在派发 Subagent 时，prompt 中显式告知项目特定的基类、命名空间、工具类
- 关键约束写在 prompt 最前面（不要埋在中间）
- 项目应维护一份 `AGENTS.md` 或类似文件，列出这些关键约定，Subagent 启动时自动加载
- 检查清单：Subagent prompt 中是否包含了——
  - 基类（`IBaseController` 而非 `MonoBehaviour`）
  - 事件系统用法（`this.SendEvent` / `this.RegisterEvent`）
  - 命名空间（`Hot.Logic` / `Hot.Logic.UI`）
  - 关键工具类（`G.userData` / `Network.Ins` 等）

### 问题 3：重复代码块未提取为公共函数

**案例：** 
- `RecoverSystem_Net.cs` 中 4 个 Done 处理器（`OnSyncRecoverHeroHpDone` / `MoodDone` / `EnergyDone` / `StrengthDone`）逻辑几乎完全相同：移除 UUID → 获取 hero → 计算变化量 → 更新属性 → 设 action → 发事件 → 获取 model → 生成 FloatingRecoverInfo。每个方法 ~20 行，90% 重复。
- `RecoverSystem_API.cs` 中 `StartRecover` 的 4 个 switch case，每个 case 都是相同模板：发网络请求 → 设置 action → 添加恢复队列 → 发银元事件 → 发开始事件。

**问题根因：** Subagent 按"每个建筑类型"线性实现，没有识别出公共模式。设计阶段也没有明确要求函数化。

**避免方法：**
- 在实现计划的 Task 描述中，如果多个 case 逻辑相同，明确要求提取公共方法
- 实现完成后，Review 阶段应当检查"是否有超过 2 处相似的代码块"
- 检查清单：Review 时目视扫描 ——
  - 同一个文件中是否有 3 个以上结构相同的代码块？
  - 多个文件中是否有相同的代码模式？
  - 如果有，合并为一个带参数的方法

### 问题 4：生成的代码缺少注释

**案例：** 本次所有新增和修改的代码（约 200 行）在上线前没有注释。最后需要额外一轮"加注释检查"来补充。`BuildingSilverFloatManager` 的节流逻辑、`RecoverSystem_API` 的队列满判断、`OnHeroSilverFlow` 的事件触发时机等都需要注释才能让后续维护者理解。

**问题根因：** Subagent 默认不写注释（遵循系统提示），但项目中非显而易见的逻辑（节流窗口设计、事件时机选择、队列预占位逻辑）确实需要注释。

**避免方法：**
- 在 Subagent prompt 中区分："不写解释代码在做什么的注释，但必须写解释 WHY 的注释"
- 如果在 Review 阶段发现缺少 WHY 注释，要求补上再通过
- 检查清单：以下类型的代码必须有注释——
  - 非直观的业务逻辑（如"为什么先本地加到队列"）
  - 设计决策（如"为什么用 0.5s 节流窗口"）
  - 特殊处理的原因（如"为什么先发事件再获取 model"）
  - 临时方案（如测试用的硬编码坐标）

## 三、Review 阶段

### 问题 5：Review 没发现关键 Bug（事件发送在 early return 之后）

**案例：** Task 4 的初始代码将银元事件发送放在 `if (controller == null) return;` 之后。如果 hero 的 3D model 未加载，事件会被静默丢失，且缓存字典中的条目泄漏。Spec reviewer 和 Code quality reviewer 都没有发现这个顺序问题。

**问题根因：** Review 更关注"代码是否符合需求描述"和"代码风格"，对边界条件（model 为 null 时事件是否仍然发出）没有深入检查。

**避免方法：**
- Review 时检查所有 early return 语句，确认 return 之前的操作和 return 之后的操作各自应该是什么
- 有副作用的代码（发事件、修改状态）应该在 early return 之前执行
- 检查清单：Review 时逐一检查 ——
  - 每个 `if (x == null) return;` 之前是否遗漏了必要的副作用？
  - 每个 `if (x == null) return;` 之后的操作是否依赖 x 非 null？
  - 字典/列表的 Add 和 Remove 是否成对出现、在所有分支都正确？

### 问题 6：Review 产生无意义争议

**案例：** Code quality reviewer 对 `OnHeroSilverFlow` 事件字段提出命名问题（PascalCase vs 文件中的 camelCase）。但这是事件类型，PascalCase 是合理的，且所有下游 Task 都引用了这些字段名。争议浪费时间且没有产生实际改进。

**问题根因：** Reviewer 缺少对"文件约定 vs 类型约定"的判断能力，机械地应用了"和周围代码一致"的规则。

**避免方法：**
- 对 Reviewer 的反馈做人工二次判断，不要无脑接受
- 如果 Reviewer 的某个建议被拒绝，在后续 Review 中应该学会这个偏好
- 事件类 / 协议类 / DTO 类的命名应遵循其自身类型的约定，不需要和所在文件的其他类保持一致

## 四、收尾阶段

### 问题 7：Git 提交污染（包含不相关的预存在修改）

**案例：** 某次 commit 包含了 ~90 行预存在的未暂存修改（`MainUIController.cs` 的 `RefreshCurrencyInfo` 重构等），这些修改与本次 feature 无关。需要通过 `git stash` → 精确 `commit` → `git stash pop` 来修复。

**问题根因：** Subagent 使用 `git add -A` 或等价操作，将工作区所有修改都加入了暂存区。没有只选择本次 feature 相关的文件。

**避免方法：**
- 提交前 `git diff --stat` 确认待提交的文件列表
- 使用 `git add <具体文件>` 而非 `git add -A` 或 `git add .`
- Subagent prompt 中明确要求："只提交与本次 Task 相关的文件，不要提交预存在的修改"
- 检查清单：提交前 ——
  - `git diff --stat` 列出的文件是否全部属于本次 feature？
  - 如果不确定，检查 `git log -1 --name-only` 看最近提交的文件范围
  - 有疑问 → `git add` 单个文件而非批量添加

## 总结：检查清单速查表

每个阶段的必查项：

| 阶段 | 必查项 |
|------|--------|
| 需求 | 时序/触发时机是否已确认？ |
| Subagent Prompt | 基类、命名空间、项目约定是否已写明？ |
| 编码 | 3+ 处相似代码是否已提取？关键逻辑是否有 WHY 注释？ |
| Review | 所有 early return 前后的副作用是否正确？ |
| 提交 | diff 文件是否全部属于本次 feature？ |
