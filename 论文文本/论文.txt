**摘 要**

在多人在线角色扮演游戏中，实时战斗系统的响应延迟与状态一致性是决定玩家体验的核心要素。传统的帧同步方案在开放大世界和强交互场景下面临反作弊能力弱、断线重连困难等挑战，而状态同步架构以服务器为唯一权威，在安全性与状态恢复方面具有天然优势。如何在状态同步框架下实现流畅的移动同步、精确的技能判定以及高效的状态广播，是实时战斗系统设计的核心问题。

本系统基于Unity引擎与.NET平台，设计并实现了一套基于状态同步的实时战斗系统原型。系统严格遵循服务器权威原则，客户端仅负责输入采集与表现渲染，所有逻辑计算均在服务端完成。在移动同步模块中，客户端采用本地预测降低操作延迟，服务端执行速度上限与障碍物校验，并通过阈值触发的位置修正策略实现多端状态和解。战斗系统实现了技能状态机、伤害计算、Buff生命周期管理及飞行物弹道模拟，技能与Buff数据均由配置表驱动，具备良好的可扩展性。背包系统采用事件驱动的增量同步机制，服务端通过StatusNotify消息推送物品变更，数据库层面结合二进制Blob与行式存储，兼顾了查询灵活性与I/O效率。系统使用Protocol Buffer定义通信协议，基于TCP长连接实现可靠的双向消息传输。

经功能验证与弱网模拟测试，本系统在典型网络环境下能够提供流畅的多人移动同步与实时战斗体验，验证了状态同步架构在MMORPG场景下的工程可行性。系统模块划分清晰、配置驱动程度高，可作为独立游戏项目的战斗底层框架或网络同步技术的教学演示平台。

关键词：状态同步；实时战斗；客户端预测；增量同步；Unity

**ABSTRACT**

In massively multiplayer online role-playing games, the responsiveness and state consistency of real-time combat systems are critical to player experience. Traditional lockstep synchronization faces challenges such as weak anti-cheat capability and difficult reconnection in open-world and highly interactive scenarios. In contrast, the state synchronization architecture, which treats the server as the sole authority, offers inherent advantages in security and state recovery. The core issues in designing a real-time combat system under the state synchronization paradigm include achieving smooth movement synchronization, precise skill hit detection, and efficient state broadcasting.

Based on the Unity engine and the .NET platform, this thesis designs and implements a real-time combat system prototype using state synchronization. The system strictly adheres to the server-authoritative principle: the client is responsible solely for input collection and presentation rendering, while all logical computations are executed on the server. In the movement synchronization module, the client employs local prediction to reduce perceived latency, the server validates speed limits and obstacle collisions, and a threshold-based position correction strategy achieves multi-client state reconciliation. The combat system implements a skill state machine, damage calculation, buff lifecycle management, and projectile trajectory simulation. Both skills and buffs are driven by configuration tables, ensuring strong extensibility. The inventory system adopts an event-driven incremental synchronization mechanism, where the server pushes item changes via StatusNotify messages, and the database employs a hybrid storage strategy combining binary Blob and row-based storage to balance query flexibility and I/O efficiency. The system uses Protocol Buffer for communication protocol definition and relies on persistent TCP connections for reliable bidirectional messaging.

Functional verification and weak network simulation tests demonstrate that the system provides smooth multi-player movement synchronization and real-time combat experience under typical network conditions, validating the engineering feasibility of the state synchronization architecture in MMORPG scenarios. The system features clear module partitioning and a high degree of configuration-driven design, making it suitable as an underlying combat framework for independent game projects or as a teaching demonstration platform for network synchronization techniques.

Key words: State Synchronization; Real-time Combat; Client-Side Prediction; Incremental Synchronization; Unity

第一章 引言

1.1 研究背景

随着互联网基础设施的持续升级和计算机图形技术的飞速发展，网络游戏产业在过去二十年中经历了从文字MUD到2D图形化，再到全3D沉浸式虚拟世界的深刻变革。根据中国音数协游戏工委发布的《2024年中国游戏产业报告》，我国游戏市场实际销售收入已突破3000亿元，其中多人在线角色扮演游戏（MMORPG）和多人竞技游戏占据了重要份额。在这一类游戏中，数十名乃至上百名玩家在同一虚拟场景中实时交互，其核心体验------移动的流畅性、战斗的打击感、操作的即时反馈------均高度依赖于底层的网络同步技术。

网络游戏同步技术旨在解决一个根本性问题：如何让地理上分散的多个客户端，对同一个虚拟世界的状态达成一致。当一名玩家在客户端按下前进键时，他的角色应当在所有其他玩家的屏幕上同步移动；当一名法师释放火球术时，伤害判定与特效播放必须在所有观看者眼中保持合理的一致性。然而，物理距离带来的网络延迟、带宽限制、数据丢包等因素，使得完美的即时同步在理论上不可能实现。因此，如何在"响应延迟"与"状态一致性"之间寻找平衡点，成为网络游戏同步技术研究的核心命题。

当前，工业界主流的同步方案可归为两类：帧同步（Lockstep）与状态同步（State Synchronization）。帧同步仅同步玩家的操作输入，各客户端以相同的初始状态和确定性逻辑独立计算游戏世界，典型代表如《星际争霸》《魔兽争霸》《王者荣耀》等RTS与MOBA类游戏。帧同步的带宽开销极小，但对客户端的计算确定性要求极高，且反作弊能力较弱，断线重连困难。状态同步则以服务器为唯一权威，所有游戏逻辑在服务端计算，客户端仅接收状态变更结果并负责表现，典型代表如《魔兽世界》《最终幻想14》《原神》等MMORPG。状态同步天然具备较强的反作弊能力，断线重连友好，但带宽消耗相对较高，且对服务端性能提出了更高要求。

在上述背景下，如何基于状态同步架构，设计并实现一套适用于实时战斗场景的完整同步系统，解决移动预测、技能判定、Buff管理等关键问题，成为本课题研究的出发点。

1.2 国内外研究现状

网络游戏同步技术的研究可追溯至20世纪90年代多人在线游戏的兴起。经过近三十年的发展，学术界与工业界在同步模型、延迟补偿、兴趣管理等方面积累了丰富成果。

帧同步技术研究方面：帧同步（Lockstep）的概念最早应用于RTS游戏。在帧同步模型中，所有客户端基于相同的初始状态和输入序列独立执行游戏逻辑计算。其核心挑战在于确定性的实现。Bettner和Terrano在《1500个弓箭手》一文中详细阐述了《帝国时代》采用定点数运算和确定性随机数生成器解决该问题的工程实践。为缓解网络延迟敏感问题，GGPO等中间件引入了"预测与回滚"（Rollback Netcode）技术，在格斗游戏中得到广泛应用。国内方面，《王者荣耀》技术团队公开分享了其基于帧同步的战斗方案，通过UDP传输、冗余发包等手段在移动网络环境下实现了较流畅的对战体验。

状态同步技术研究方面：状态同步是MMORPG的主流方案。其核心思想是服务端作为游戏世界的唯一真实来源，客户端仅负责表现。研究热点包括：客户端预测与和解（Gabriel Gambetta系统阐述，并被Valve Source引擎、《守望先锋》等采用）、兴趣区域管理（Boulanger等人对各类AOI算法进行了性能对比分析）、增量同步与状态压缩（如Quake 3的Delta压缩、Protocol Buffer变长编码）。国内学术界方面，北京邮电大学、电子科技大学等高校的硕士学位论文针对FPS游戏的命中判定同步、MMORPG的技能同步进行了系统设计。工业界方面，《原神》技术团队介绍了其基于状态同步的"多人世界"架构；《永劫无间》则在动作游戏中融合了状态同步与帧同步的思想。

现状总结：帧同步适合固定参与者、短对局时长的竞技游戏；状态同步在反作弊、断线重连、大世界支持方面更具优势，适合MMORPG及开放世界游戏。然而，状态同步在实时战斗场景下的实现复杂度较高，且目前国内高校相关论文多为单一模块的实现，缺乏覆盖移动、技能、Buff、背包的完整实时战斗系统的设计与论述。

本课题旨在设计并实现一套基于状态同步架构的实时战斗系统原型，重点解决状态同步框架下客户端预测、服务器和解、增量广播等关键技术问题，构建可玩、可控、可扩展的实时战斗系统。

第二章 相关理论与技术基础

2.1 状态同步原理

状态同步（State Synchronization）是一种广泛应用于多人在线游戏的网络同步架构。其核心思想是：服务端作为游戏世界的唯一权威，负责所有逻辑计算与状态变更；客户端仅作为输入采集器与表现渲染器，接收服务端下发的状态数据并呈现给玩家。

状态同步的工作流程可概括为以下四个步骤：

1. 客户端采集输入：玩家通过键盘、鼠标等外设产生的操作（如移动、释放技能），被客户端封装为操作指令发送至服务端。

2. 服务端执行逻辑：服务端将客户端指令放入逻辑帧队列，在固定时间步长内依次执行，更新受影响的实体状态（位置、属性、Buff等）。此过程中服务端会进行合法性校验，拒绝任何作弊或异常操作。

3. 状态广播：服务端将本帧内发生变更的实体状态序列化为同步消息，广播给视野内所有相关客户端。

4. 客户端表现：客户端接收同步消息后，更新本地数据模型，并驱动角色动画、UI等表现层元素。

2.2 帧同步与状态同步对比分析

帧同步（Lockstep）与状态同步（State Synchronization）是当前工业界最为通用的两种网络同步方案，二者在适用场景、实现复杂度、安全性等方面各有优劣。

2.2.1 帧同步

帧同步方案的核心思想是：各客户端以相同的初始状态启动，基于相同的输入序列独立执行确定性游戏逻辑计算，从而得到一致的最终状态[1][5]。其优势在于带宽开销极低——仅需同步玩家的操作指令而非完整的游戏状态数据。然而，帧同步存在以下不足：第一，要求游戏逻辑必须是确定性的（Deterministic），即相同的输入在任何平台、任何时间执行都必须产生完全相同的输出。这在涉及浮点运算、物理模拟时极难保证。第二，反作弊能力弱，因为所有逻辑均在客户端计算，恶意客户端可直接修改内存数据。第三，断线重连困难，新加入的客户端需要从服务端拉取完整的历史输入序列进行回溯计算。

帧同步适用于固定参与者、短对局时长、强实时性的竞技游戏，如RTS和MOBA类游戏[3][8]。

2.2.2 状态同步

状态同步则以服务端为唯一权威（Server Authority），所有游戏逻辑在服务端完成计算，客户端仅负责输入采集与表现渲染[2][10]。服务器接收到客户端输入后执行合法性校验，更新权威游戏状态，然后将状态变更广播给所有相关客户端。其核心优势包括：第一，天然具备较强的反作弊能力，恶意客户端无法绕过服务端校验直接修改游戏状态。第二，断线重连友好，重连客户端可从服务端获取当前完整的游戏状态快照。第三，对客户端的计算能力要求较低，适合移动端等资源受限平台。代价是服务端计算压力较大，且带宽消耗相对较高[14]。

状态同步适合玩家数量较多、游戏世界较大、对安全性和持久性要求较高的MMORPG及开放世界游戏。

2.2.3 对比总结

综合上述分析，帧同步与状态同步的核心差异可归纳为下表：

表2-1 帧同步与状态同步对比

| 对比维度     | 帧同步           | 状态同步         |
|------------|-----------------|-----------------|
| 同步内容     | 玩家输入指令       | 游戏状态数据       |
| 服务器职责   | 转发输入           | 逻辑计算+权威裁决  |
| 客户端职责   | 独立逻辑计算+表现   | 仅表现渲染         |
| 确定性要求   | 极高              | 无               |
| 反作弊能力   | 弱               | 强               |
| 断线重连难度 | 高               | 低               |
| 带宽消耗     | 低               | 相对较高           |
| 适用游戏类型 | RTS、MOBA、格斗   | MMORPG、开放世界   |

鉴于本课题所设计的是MMORPG场景下的实时战斗系统，对安全性、状态恢复能力和可扩展性有较高要求，因此选择状态同步作为系统的核心同步方案[15]。

2.3 开发环境与工具链

本系统开发涉及客户端、服务端及通信协议三个部分，所采用的技术栈如下：

2.3.1 客户端技术栈

- 游戏引擎：Unity 2020.3 LTS。Unity提供完善的组件化开发框架、物理引擎（Rigidbody）、动画系统（Animator）及UI框架（UGUI），能够快速搭建游戏原型并支持跨平台发布。

- 开发语言：C#。Unity原生支持的脚本语言，语法现代、内存管理便捷，与服务端语言保持一致，便于代码逻辑的对称实现。

- 网络通信：基于 .NET Socket 实现 TCP 长连接客户端 NetClient，负责与游戏服务器建立可靠连接，处理粘包、断线重连等底层细节。

- 序列化库：Google Protocol Buffer（protobuf-net库）。将结构化的协议对象序列化为紧凑的二进制流，在保证跨语言兼容性的同时显著降低传输数据量。

2.3.2 服务端技术栈

- 运行环境：.NET Framework 4.7.2 / .NET Core 3.1。提供稳定、高性能的托管运行时，支持异步 I/O 操作。

- 开发语言：C#。与客户端语言统一，降低学习成本，便于技能逻辑、伤害公式等代码在两端保持对称。

- 数据持久化：SQL Server。用于存储玩家账号、角色属性、背包物品等持久化数据。背包等大数据量字段采用 varbinary(MAX) 存储 Protobuf 序列化后的字节流，减少 I/O 次数。

- 定时与日志：服务端实现逻辑帧驱动的主循环，利用高精度计时器驱动 Map 与 Battle 对象的 Update 方法。日志系统记录玩家关键操作与异常信息，便于调试与运营分析。

2.3.3 通信协议与工具

- 协议定义：使用 .proto 文件统一定义所有网络消息结构，通过 protoc 编译器生成 C# 类文件，保证客户端与服务端协议严格一致。

- 抓包调试工具：Wireshark 用于分析 TCP 数据流，验证协议封包格式及同步频率。

- 网络模拟工具：Clumsy 用于模拟高延迟、丢包、乱序等弱网环境，测试系统的鲁棒性与位置修正策略的有效性。

2.4 设计模式应用概述

在系统设计与实现过程中，合理运用设计模式能够提高代码的可复用性、可维护性与扩展性。本系统中主要应用了以下设计模式：

1. 单例模式（Singleton Pattern）

系统中的全局管理器均采用单例模式实现，如客户端 Managers 命名空间下的 EntityManager、BattleManager、ItemManager，服务端的 MapManager、EntityManager、BattleManager 等。单例模式确保这些管理器在应用程序生命周期内仅有一个实例，提供全局统一的访问入口，便于管理实体列表、配置数据及网络服务。

实现上，通过泛型基类 Singleton<T> 封装线程安全的懒加载逻辑，派生类无需重复编写单例样板代码。

2. 观察者模式（Observer Pattern）

观察者模式用于解耦事件发布者与订阅者。在本系统中，主要体现在以下场景：

- UI 与数据模型：当背包物品数量变化时，ItemManager 作为被观察者，触发 OnItemChange 事件，UIBag 等 UI 组件作为观察者订阅该事件并刷新显示。

- 网络消息分发：服务端 MessageDistributer 维护消息类型与处理委托的映射表，网络层接收到消息后，通知已注册的业务服务（如 MapService.OnMapEntitySync），实现网络层与业务逻辑的解耦。

- 实体状态变更：EntityManager 允许外部组件注册 IEntityNotify 接口，当实体发生位置变化或事件（如移动、跳跃）时，主动通知对应的 EntityController 更新表现。

3. 状态模式（State Pattern）

状态模式在技能系统和 Buff 系统中得到了明确应用。

- 技能状态机：Skill 类内部维护 SkillStatus 枚举（None、Casting、Runing），在不同状态下对 OnUpdate 方法的响应逻辑不同。例如，Casting 状态仅累加吟唱计时，达到吟唱时间后自动切换到 Runing 状态。状态模式将复杂的分支逻辑封装在各个状态对应的处理函数中，使技能主循环代码简洁清晰。

- Buff 效果管理：BuffEffect 枚举定义了眩晕、沉默、减速等多种效果类型。EffectManager 维护当前生效的效果集合，角色在尝试移动或释放技能时，通过查询 HasEffect 来决定行为是否被禁止。这种设计使得新增控制效果时无需修改角色主逻辑，仅需扩展效果枚举及对应处理即可。

4. 组件模式（Component Pattern）

Unity 引擎本身即基于组件模式构建，本系统在此基础上进一步应用：

- 玩家角色对象挂载了 PlayerInputController（输入采集）、EntityController（状态表现）、RideController（坐骑控制）等多个脚本组件，各组件职责单一，通过 Unity 消息函数（Start、FixedUpdate）协作。

- 服务端虽未使用 Unity 引擎，但在设计上模拟了组件思想：Creature 对象聚合了 SkillManager、BuffManager、EffectManager 等子管理器，每个子管理器专注于特定领域的逻辑，Creature 仅作为协调者调用它们的 Update 方法。

5. 工厂模式（Factory Pattern）

工厂模式用于封装对象创建逻辑。例如：

- 客户端 GameObjectManager 负责根据配置 ID 加载角色、怪物、坐骑的预制体并实例化，隐藏了资源加载与对象池管理的细节。

- 服务端 EntityManager.AddEntity 方法根据实体类型（玩家、NPC、怪物）创建对应的 Entity 派生类实例，并分配唯一的 entityId。

6. 策略模式（Strategy Pattern）

策略模式用于算法族的选择与替换。本系统中，伤害计算可视为策略模式的一种体现：

- 技能配置中的 TargetType（自身、敌方单体、位置范围）决定了命中判定的策略。Skill.DoHit 方法根据配置的 AOERange 和 CastTarget 动态选择是调用 HitTarget 直接攻击单体，还是通过 HitRange 遍历范围内的所有目标。

第三章 系统需求分析

3.1 功能性需求分析

依据业务场景，系统功能可划分为背包与属性管理、移动同步、战斗功能三大模块。以下对各模块进行详细需求分析。

3.1.1 背包与角色属性管理需求

本部分功能为战斗系统提供数值基础与道具交互支持，是玩家角色成长的外围保障。

1. 背包与道具管理

- 物品增删改查：系统应支持物品的获得、使用、丢弃、出售等操作。所有物品变更必须由服务端验证并持久化，客户端仅在收到服务端确认后更新UI。

- 物品堆叠与排序：同类可堆叠物品应能自动合并显示，背包界面应支持按类型、品质等条件排序。

- 装备穿戴与卸下：玩家可将背包中的装备穿戴至对应槽位，装备属性应实时累加至角色战斗属性。穿戴操作需校验职业、等级等限制条件。

- 商店交易：玩家可与NPC商店进行物品买卖，交易需校验玩家金币余额及商店库存。

2. 角色属性管理

- 基础属性：包括等级、经验值、生命值（HP）、法力值（MP）、物理攻击（AD）、法术攻击（AP）、物理防御（DEF）、法术防御（MDEF）、暴击率（CRI）等。

- 属性来源与计算：角色最终属性应由基础属性（等级成长）、装备属性、Buff属性三者线性叠加而成。属性变更后需同步至客户端UI。

- 升级与经验：玩家通过战斗获得经验值，经验值达到阈值后自动升级，升级后应回复满状态并解锁新技能或提升技能等级上限。

3.1.2 移动同步功能需求

移动同步是实时战斗系统的底层基础，需保证多个玩家在同一地图中位置、朝向、动作的一致性。

1. 玩家输入采集与本地响应

- 客户端应实时采集键盘输入（W/A/S/D移动，鼠标/键盘旋转视角，空格跳跃）。

- 客户端应立即对输入做出本地移动响应，以降低视觉延迟，提升操作手感。

- 当玩家输入导致状态变化（静止→移动、移动→静止、跳跃）时，客户端应立即向服务端发送同步请求。

2. 服务端移动校验与广播

- 服务端接收客户端上报的移动状态后，需校验移动合法性（速度是否超过角色最大移速、是否穿越障碍物、垂直位移是否符合重力规则等）。

- 校验通过后，服务端更新权威实体位置，并向视野内所有客户端广播状态变更。

- 对于非法移动（如瞬移、加速），服务端应拒绝更新并将玩家拉回上一合法位置。

3. 客户端位置修正

- 客户端在接收服务端广播的自身位置后，应比对本地预测位置与服务端权威位置的偏差。

- 若偏差超过预设阈值（如50逻辑单位），客户端应立即将本地位置强制同步至服务端位置，避免长期偏离。

3.1.3 战斗功能需求

战斗功能是系统的核心玩法，涵盖技能释放、伤害计算、Buff管理、飞行物模拟等子功能。

1. 技能管理与释放

- 技能列表：每个角色拥有一组技能，技能数据包含等级、冷却时间、法力消耗、施法距离、伤害系数等配置。

- 释放校验：玩家释放技能时，客户端应进行本地预校验（冷却、法力、距离），通过后发送请求至服务端。服务端需进行二次权威校验，防止客户端篡改。

- 技能状态机：技能应具备空闲（None）、吟唱（Casting）、执行中（Runing）三种状态。吟唱类技能需经过吟唱时间后方可生效；引导类技能在持续时间内每间隔固定时间造成一次伤害。

- 冷却与资源消耗：技能释放成功后，服务端应扣除相应法力值并进入冷却状态，冷却时间结束后方可再次释放。

2. 伤害计算与命中判定

- 目标类型支持：技能需支持自身、敌方单体、位置范围、方向矩形等多种目标类型。

- 伤害公式：伤害计算应综合考虑施法者攻击力、目标防御力、技能基础伤害及系数，支持物理与法术双属性分别计算减免。伤害结果应包含暴击判定及随机浮动。

- 范围判定：对于AOE技能，服务端应根据技能配置的范围参数，查询范围内所有合法目标并分别计算伤害。

- 伤害广播：服务端应将本次技能造成的所有伤害数据封装为 SkillHitResponse 广播给相关客户端，客户端据此更新目标血量并显示伤害数字。

3. Buff系统

- Buff类型：支持增益（增加攻击力、防御力、回血等）和减益（眩晕、沉默、持续掉血等）效果。

- Buff生命周期：Buff应有明确的持续时间，到期后自动移除。持续性效果（如每秒回血）应按固定间隔触发。

- Buff叠加规则：同一Buff可配置为不可叠加（刷新时间）、独立叠加层数、或不同施法者独立计算。

- Buff同步：服务端在Buff添加、移除、触发伤害时，应向客户端发送 BuffResponse 消息，客户端据此更新UI（Buff图标、剩余时间）和角色状态。

4. 飞行物系统

- 弹道技能支持：对于弓箭、火球等投射物技能，系统应支持飞行物对象，在技能释放时创建，飞行至目标后触发伤害。

- 服务端飞行物：服务端采用时间驱动模式，根据距离与速度计算飞行时长，到期后执行伤害结算。

- 客户端飞行物：客户端创建对应的飞行物特效，根据起始点和目标点插值运动，在服务端伤害消息到达时播放命中特效。

5. 战斗状态管理

- 角色应具备战斗状态（是否处于战斗中），进入战斗后生命/法力恢复速度降低或暂停，脱离战斗一定时间后重置。

- 战斗状态下，技能释放、道具使用等行为可能受到限制（如部分道具无法在战斗中使用）。

3.2 非功能性需求分析

本系统在满足功能需求的同时，还需满足以下非功能性需求：

1. 实时性：移动同步消息从客户端发送到服务端再广播至其他客户端的往返延迟应小于200ms，以保证玩家操作手感流畅[1][2]。技能从释放到命中的响应时间也应控制在可接受范围内。

2. 安全性：系统必须严格遵循服务端权威原则，所有涉及数值变更的逻辑操作必须在服务端完成最终校验，杜绝客户端作弊行为（如加速、瞬移、属性篡改等）[10]。

3. 可扩展性：技能与Buff的数值和行为应由配置表驱动，新增技能类型或Buff效果时不应修改核心代码逻辑，仅需扩展配置数据。系统模块划分应支持独立开发和测试[12]。

4. 可靠性：系统应支持断线重连功能，玩家在短时间内断线后重新连接，应能从服务端恢复其完整的游戏状态（位置、属性、背包等）[13]。

5. 并发支持：服务端应支持同一地图内至少50名玩家同时在线，并能在百人同屏场景下保持稳定的逻辑帧率（不低于20帧/秒）。

第四章 系统总体设计

4.1 系统整体架构设计

本系统采用经典的客户端---服务器（C/S）分层架构，并严格遵循服务端权威（Server Authoritative）的设计原则。客户端负责采集玩家输入、渲染游戏画面、播放音效与特效，所有与游戏逻辑相关的数值计算、状态变更、合法性校验均在服务端完成，客户端仅作为表现终端与服务端状态的订阅者。整体架构如图4-1所示。

[图4-1 系统整体架构图]

各层职责如下：

- 客户端表现层：基于 Unity 引擎的 UGUI 系统实现，包括背包界面（UIBag）、技能栏（UISkill）、角色状态面板等。该层订阅逻辑层的数据变更事件，仅负责将数据可视化。

- 客户端逻辑层：维护与表现层解耦的数据模型，如 Creature、Item、Skill 等。同时负责采集输入（PlayerInputController）并向服务端发送操作请求。

- 客户端网络层：封装 TCP 长连接，负责 Protobuf 消息的序列化、反序列化以及粘包处理，向上层提供异步消息回调接口。

- 服务端网络门面层：基于 MessageDistributer 模式分发客户端请求至对应的业务服务（MapService、BattleService、ItemService 等）。

- 服务端业务逻辑层：系统的核心，包含移动同步逻辑（Map、Creature）、战斗逻辑（Battle、Skill、Buff）、背包与道具管理（ItemManager）等功能模块。该层直接操作内存中的实体对象。

- 服务端数据访问层：负责实体状态与关系数据库（SQL Server / MySQL）之间的持久化映射，采用 ORM 与原始 SQL 混合方式。

- 基础设施层：提供跨模块使用的通用功能，如单例模板（Singleton<T>）、日志系统（Log）、定时器、配置文件读取（DataManager）等。

4.2 功能模块划分

依据高内聚、低耦合的软件设计原则，系统从功能维度划分为三个核心子系统：背包系统、移动同步系统和战斗系统。各子系统之间的关系如图4-2所示。

[图4-2 系统功能模块划分图]

4.2.1 背包系统

背包系统为游戏提供道具存储、装备穿戴与属性累加等基础功能，是玩家角色成长与战斗数值的重要外围支撑。

- 物品增删管理：支持物品的获得、使用、丢弃、出售等操作。客户端 ItemManager 与服务端 ItemManager 通过 StatusNotify 消息进行状态同步，采用事件驱动模式实现 UI 局部刷新。

- 装备穿戴与属性：管理角色装备栏位的穿戴与卸下，装备属性会动态累加至角色战斗属性（Attributes.Equip）中，直接影响技能伤害与防御计算。

- 事件驱动增量同步：背包数据在数据库中采用二进制 Blob 与行式存储结合的混合策略，平衡了查询灵活性与 I/O 效率。服务端将物品变更封装为增量消息推送至客户端，避免频繁全量拉取。

4.2.2 移动同步系统

移动同步是实时战斗的底层支撑，负责将玩家的位置、朝向、速度等状态在多客户端间保持一致。本系统采用服务器权威的增量同步策略，其核心组件包括：

- 客户端 PlayerInputController：采集键盘输入，执行本地预测移动，并通过 MapService 上报 MapEntitySyncRequest。

- 客户端 EntityController：接收服务端广播的实体状态，驱动动画与 Transform 更新。

- 服务端 Map / MapManager：管理地图内所有实体，在逻辑帧中更新实体位置并执行合法性校验（速度上限、障碍物碰撞等）。

- AOI（Area of Interest）模块：基于网格或十字链表过滤视野，仅向玩家广播周围实体的状态变更，降低带宽消耗。

移动同步的详细设计与位置修正策略将在5.2节中展开。

4.2.3 战斗系统

战斗系统实现了技能释放、伤害计算、Buff生命周期管理及飞行物弹道模拟，是游戏可玩性的核心。该系统同样遵循服务端权威原则，客户端仅负责请求发送与结果表现。

- 技能管理模块（Skill / SkillManager）：管理技能的冷却、吟唱、多段打击时间轴，客户端与服务端共享相同的技能状态机逻辑以确保表现一致性。

- 伤害计算模块：采用"攻击力---防御减免"公式，支持物理、法术双属性，包含暴击、随机浮动等机制。

- Buff 管理模块（Buff / BuffManager）：处理增益/减益效果的添加、计时、属性修正与移除，通过 BuffResponse 消息同步客户端 UI。

- 飞行物模块（Bullet）：服务端采用时间驱动模式延迟伤害判定，客户端并行模拟弹道运动以提供流畅视觉反馈。

- 战斗管理器（Battle / BattleManager）：维护参与战斗的单位集合，驱动每帧技能更新并聚合战斗消息进行批量广播。

战斗系统的详细设计与实现将在5.3节中深入阐述。

4.4 通信协议设计

本系统网络通信基于 TCP 长连接，应用层协议采用 Google Protocol Buffer 进行消息序列化。协议定义文件为 message.proto，统一管理所有请求与响应结构。

4.4.1 消息包结构

底层传输采用自定义的长度前缀 + Protobuf 数据的封包格式：

[ 4字节消息长度 (int32) ][ Protobuf 序列化后的 NetMessage 字节流 ]

NetMessage 为顶层消息容器，定义如下：

message NetMessage {
    NetMessageRequest Request = 1;
    NetMessageResponse Response = 2;
}

一次通信要么是客户端发起的 Request，要么是服务端主动下发的 Response，二者互斥。

4.4.2 状态同步核心协议

移动同步与实体状态变更通过 NEntitySync 消息承载：

message NEntity {
    int32 id = 1;
    NVector3 position = 2;
    NVector3 direction = 3;
    int32 speed = 4;
}

message NEntitySync {
    int32 id = 1;
    ENTITY_EVENT event = 2;
    NEntity entity = 3;
    int32 param = 4;
}

MapEntitySyncRequest 用于客户端上报自身状态，MapEntitySyncResponse 用于服务端广播视野内所有实体的状态变更。

4.4.3 战斗相关协议

战斗协议包含技能施法、命中、Buff三类核心消息：

message SkillCastRequest {
    NSkillCastInfo castInfo = 1;
}

message SkillCastResponse {
    RESULT result = 1;
    repeated NSkillCastInfo CastInfoes = 2;
}

message SkillHitResponse {
    RESULT result = 1;
    repeated NSkillHitInfo Hits = 2;
}

message BuffResponse {
    RESULT result = 1;
    repeated NBUFFInfo Buffs = 2;
}

NSkillCastInfo 携带技能 ID、施法者、目标、释放位置及施法结果；NSkillHitInfo 包含本次命中造成的所有伤害详情；NBUFFInfo 描述 Buff 的增删及持续伤害跳字。

4.4.4 背包及状态通知协议

背包等非高频状态变更采用统一的 StatusNotify 消息进行增量推送：

message StatusNotify {
    repeated NStatus status = 1;
}

message NStatus {
    STATUS_TYPE type = 1; // MONEY, EXP, LEVEL, ITEM, SKILL...
    STATUS_ACTION action = 2; // UPDATE, ADD, DELETE
    int32 id = 3;
    int32 value = 4;
}

当玩家获得物品、金币变动、经验增加时，服务端 StatusManager 会构造 NStatus 对象并立即推送给客户端。客户端 StatusService 解析后分发至对应的管理器（如 ItemManager.OnItemNotify），驱动 UI 局部刷新。该设计避免了每次背包变动都拉取全量数据，有效节省带宽。

第五章 系统详细设计与实现

5.1 背包系统

背包系统不仅是物品数据的容器，更是验证状态同步机制在非高频、强一致性场景下应用的典型案例。本系统采用服务端权威架构，确保玩家虚拟资产的安全性与多端一致性。

5.1.1 数据存储与同步策略

服务端对物品数据采用混合存储策略，以兼顾查询灵活性与读写性能。

1. 行式存储

如表 TCharacterItems 所示，每一条物品记录独立成行，包含字段：Id（主键）、CharID（所属角色）、ItemID（物品配置ID）、Count（数量）、Position（背包槽位）等。该方案便于对特定物品（如任务道具）进行 SQL 条件查询，但物品数量较多时读写数据库频率较高。

2. 二进制 Blob 存储

如表 CharacterBags 所示，将整个背包序列化为 varbinary(MAX) 字段。在代码实现中，利用 C# 的 unsafe 指针操作将 BagItem[] 结构体直接映射为字节流：

public unsafe byte[] SerializeBag(BagItem[] items) {
    byte[] data = new byte[items.Length * sizeof(BagItem)];
    fixed (byte* ptr = data) {
        for (int i = 0; i < items.Length; i++) {
            *(BagItem*)(ptr + i * sizeof(BagItem)) = items[i];
        }
    }
    return data;
}

此方法在网络传输及数据库写回时，仅需一次 I/O 操作，显著降低了状态同步的开销。

5.1.2 状态变更的订阅与推送机制

为避免客户端轮询造成的带宽浪费，本模块实现了基于事件驱动的增量同步。

服务端核心逻辑位于 GameServer.Managers.ItemManager：

public bool AddItem(int itemID, int count) {
    // 1. 更新内存权威数据
    item.Add(count);
    // 2. 标记数据库变更
    // 3. 触发状态同步事件
    Owner.StatusManager.AddItemChange(itemID, count, StatusAction.Add);
    return true;
}

StatusManager 将变更封装为 StatusNotify 消息，通过已建立的网络会话推送给特定玩家客户端。

客户端响应逻辑位于 Managers.ItemManager：

private bool OnItemNotify(NStatus status) {
    if (status.Action == StatusAction.Add)
        this.AddItem(status.Id, status.Value);
    // ...
}

客户端 ItemManager 作为订阅者，接收到推送后直接修改本地 Model 层数据（BagManager 与 ItemManager 字典），随后利用 Unity 的事件中心驱动 UI 层的 UIBag 进行局部刷新。

5.1.3 同步机制在背包交互中的应用

[图5-1 购买物品状态同步时序图]

以玩家购买商店道具为例，完整的状态同步时序如图5-1所示：

1. 客户端点击购买按钮，发送 ItemBuyRequest。
2. 服务端校验金币余额与商店库存，扣除金币并添加物品至背包。
3. 服务端 StatusManager 生成 StatusNotify 消息，向客户端推送物品增加及金币减少事件。
4. 客户端接收推送，更新本地背包数据与金币显示，并播放获得物品特效。

在整个流程中，客户端在发送请求后并不立即乐观更新 UI，而是等待服务端的 Response 或 Notify 回包后再进行表现。这种"请求-验证-广播-表现"的模式，从根本上杜绝了客户端作弊修改物品数量的可能，也保证了在弱网环境下，断线重连后客户端能从服务端拉取到准确的背包全量快照。

5.2 移动同步系统

移动同步是实时战斗系统的基础支撑，也是状态同步架构中最具代表性的实现案例。本系统采用服务器权威的同步模型，客户端负责采集玩家输入并进行本地预测渲染，服务端负责验证移动合法性并向视野内所有客户端广播状态变更。

5.2.1 移动同步协议设计

移动同步的数据交换基于 Protocol Buffer 定义的消息结构。核心消息体如下：

message NEntity {
    int32 id = 1;
    NVector3 position = 2;
    NVector3 direction = 3;
    int32 speed = 4;
}

message NEntitySync {
    int32 id = 1;
    ENTITY_EVENT event = 2;
    NEntity entity = 3;
    int32 param = 4;
}

message MapEntitySyncRequest {
    NEntitySync entitySync = 1;
}

message MapEntitySyncResponse {
    repeated NEntitySync entitySyncs = 2;
}

各字段含义如下：

- NEntity：封装实体的核心运动状态，包括唯一标识符 id、逻辑坐标 position、朝向向量 direction 以及当前移动速度 speed。所有坐标均采用整型定点数，以消除浮点数在不同平台间的精度差异。

- NEntitySync：同步消息的原子单元。event 字段标识触发同步的行为类型（如 MOVE_FWD、JUMP、IDLE 等）；entity 携带更新后的实体状态；param 用于传递额外参数。

- MapEntitySyncRequest：客户端上报自身状态变更的请求消息。

- MapEntitySyncResponse：服务端广播视野内实体状态变更的响应消息，支持批量同步以降低网络包数量。

该协议设计遵循"事件驱动 + 状态全量"的混合模式：当玩家操作触发状态转换时，客户端立即发送携带完整实体状态的同步请求；服务端校验后将状态变更广播给所有观察者。

5.2.2 客户端移动同步实现

客户端移动同步由 PlayerInputController 和 EntityController 两个核心组件协同完成。

1. 输入采集与本地预测

PlayerInputController 挂载于玩家角色对象，在 FixedUpdate 方法中以固定物理帧率采集输入：

float v = Input.GetAxis("Vertical");
float h = Input.GetAxis("Horizontal");
if (v > 0.01) {
    state = CharacterState.Move;
    this.character.MoveForward();
    this.rb.velocity = this.rb.velocity.y * Vector3.up +
        GameObjectTool.LogicToWorld(character.direction) * (this.character.speed + 9.81f) / 100f;
}

为降低视觉延迟，客户端在接收到输入后立即修改本地刚体速度，而不等待服务端确认。这种"客户端预测"机制使得玩家操作与画面反馈之间的间隔压缩至一帧以内。

当玩家状态发生变更时，SendEntityEvent 方法被调用：

public void SendEntityEvent(EntityEvent entityEvent, int param = 0) {
    if (entityController != null)
        entityController.OnEntityEvent(entityEvent, param);
    MapService.Instance.SendMapEntitySync(entityEvent, this.character.EntityData, param);
}

该方法一方面触发本地动画（通过 EntityController.OnEntityEvent），另一方面将 NEntitySync 消息发送至服务端。

2. 远程实体状态应用

对于非玩家角色，客户端完全信任服务端下发的状态。EntityManager 接收 MapEntitySyncResponse 后，通过注册的 IEntityNotify 接口通知对应的 EntityController：

internal void OnEntitySyncs(NEntitySync data) {
    entities.TryGetValue(data.Id, out Entity entity);
    if (entity != null) {
        if (data.Entity != null) entity.EntityData = data.Entity;
        if (notifiers.ContainsKey(data.Id)) {
            notifiers[entity.entityId].OnEntityChanged(entity);
            notifiers[entity.entityId].OnEntityEvent(data.Event, data.Param);
        }
    }
}

EntityController.OnEntityChanged 负责将逻辑坐标转换为世界坐标并更新 Transform：

public void UpdateTransform() {
    this.position = GameObjectTool.LogicToWorld(entity.position);
    this.direction = GameObjectTool.LogicToWorld(entity.direction);
    this.rb.MovePosition(this.position);
    this.transform.forward = this.direction;
}

采用 Rigidbody.MovePosition 而非直接赋值 transform.position，是为了保持物理引擎的连续性与碰撞检测的正确性。

5.2.3 服务端移动同步实现

服务端作为移动逻辑的最终裁决者，承担输入校验、状态更新与广播三项职责。

1. 请求接收与分发

客户端上报的 MapEntitySyncRequest 由 MapService 处理：

private void OnMapEntitySync(NetConnection<NetSession> sender, MapEntitySyncRequest request) {
    Character character = sender.Session.Character;
    MapManager.Instance[character.Info.mapId].UpdateEntity(request.entitySync);
}

服务端从会话中获取玩家角色对象，并根据其所在地图 ID 将同步请求转发至对应的地图实例。

2. 移动合法性校验

服务端对客户端上传的位置并非无条件采信，而是进行严格的合法性校验，典型校验项包括：

- 位置变化量是否超过理论最大位移（速度 × 时间间隔）。
- 是否穿越了不可行走区域（如障碍物、水域）。
- 垂直方向位移是否符合重力规则。

校验通过后，服务端更新权威实体状态，并将其标记为"脏数据"，待下一轮广播周期发送给视野内客户端。

3. 状态广播

MapService.SendEntityUpdate 方法将变更的实体状态封装为 MapEntitySyncResponse 并发送至指定连接：

internal void SendEntityUpdate(NetConnection<NetSession> conn, NEntitySync entitySync) {
    conn.Session.Response.mapEntitySync = new MapEntitySyncResponse();
    conn.Session.Response.mapEntitySync.entitySyncs.Add(entitySync);
    conn.SendResponse();
}

在实际运行中，广播会经过兴趣区域（AOI）过滤，仅向视野内包含该实体的客户端发送更新，以节省带宽。

5.2.4 客户端位置修正策略

由于网络延迟的存在，客户端本地预测的位置与一段时间后到达的服务端权威位置必然存在偏差。为处理这种不一致，本系统在 PlayerInputController.LateUpdate 中实现了周期性位置和解：

if ((GameObjectTool.WorldToLogic(this.rb.transform.position) - this.character.position).magnitude > 50) {
    this.character.SetPosition(GameObjectTool.WorldToLogic(this.rb.transform.position));
    this.SendEntityEvent(EntityEvent.None);
}

朝向同步也采用类似的阈值判定（偏差超过 10° 时触发修正）。当偏差超过预设阈值时，客户端立即将本地逻辑状态同步至表现层，并上报一次静默同步事件。这种"阈值触发"的和解策略在保证最终一致性的同时，避免了因高频微小修正导致的画面抖动，是状态同步中平衡响应性与平滑性的经典做法。

5.2.5 移动同步时序总结

[图5-2 移动同步时序图]

综合上述流程，一次完整的玩家移动同步交互时序如图5-2所示：

1. 客户端采集输入，执行本地预测移动。
2. 客户端发送 MapEntitySyncRequest 至服务端。
3. 服务端校验移动合法性，更新权威状态。
4. 服务端广播 MapEntitySyncResponse 给视野内所有客户端。
5. 其他客户端接收广播，更新对应角色的位置与动画。
6. 玩家自身客户端接收广播后，若本地位置与权威位置偏差超过阈值，执行位置修正。

通过客户端预测、服务端校验、广播同步、阈值修正四层机制，本系统在典型网络环境（延迟 < 100ms）下实现了流畅且一致的多人移动同步体验。

5.3 战斗系统

战斗系统是实时战斗的核心功能模块，负责处理技能释放、伤害计算、Buff（增益/减益效果）生命周期管理以及飞行物弹道模拟。本系统严格遵循服务器权威原则：所有战斗逻辑均在服务端执行，客户端仅负责发送操作请求并表现服务端下发的战斗结果。

5.3.1 战斗系统总体架构

[图5-3 战斗系统架构]

战斗系统的整体架构如图5-3所示。客户端通过 BattleService 发送 SkillCastRequest 至服务端；服务端 BattleService 接收请求后转交 BattleManager，再由当前地图的 Battle 实例进行处理。Battle 对象维护参与战斗的所有单位（AllUnits），并在每帧的 Update 中驱动技能状态更新、Buff 计时、飞行物运动以及伤害广播。

Battle 类内部维护了三个消息队列：CastSkills（施法信息）、Hits（命中伤害）、BuffActions（Buff增减），每帧结束后统一广播给地图内的所有客户端，实现战斗状态的批量同步。

5.3.2 技能系统设计

技能系统由客户端 Battle.Skill 和服务端 GameServer.Battle.Skill 对称实现，两端的核心逻辑保持高度一致，以确保客户端预测表现与服务端裁决结果尽可能吻合。

1. 技能数据结构与状态机

每个技能由 SkillDefine（配置表数据）和 NSkillInfo（玩家技能等级信息）共同描述。技能配置包含以下关键字段：

- CastTarget：目标类型（自身、敌方单体、位置范围）。
- CastRange：释放距离。
- CastTime：吟唱时间。
- Duration：技能持续时间。
- Interval：持续伤害/治疗间隔。
- HitTimes：多段打击的时间点数组。
- Bullet：是否为飞行物技能。
- AD/AP 及系数：物理/法术伤害基础值与加成比例。
- MPCost、CD：资源消耗与冷却时间。

技能状态机定义了三种状态：None（空闲）、Casting（吟唱中）、Runing（执行中）。状态转换条件如下：

- 玩家发起施法请求，若校验通过且 CastTime > 0，进入 Casting 状态。
- 吟唱计时达到 CastTime 后，转入 Runing 状态。
- 技能持续时间结束或所有打击次数完成后，回到 None 状态。

2. 技能释放流程与合法性校验

服务端 Skill.CanCast 方法执行严格的释放前校验，校验项包括：

- 技能是否处于可释放状态（Status == SkillStatus.None）。
- 目标是否合法（单体技能必须有有效目标，且目标不能为自身）。
- 施法距离是否在 CastRange 内。
- 法力值是否足够。
- 技能冷却是否已结束。

校验通过后，Skill.Cast 方法初始化冷却时间、吟唱计时、打击计数等状态。若技能为瞬时释放，则立即调用 DoHit 造成伤害；否则进入 Casting 或 Runing 状态等待后续帧更新。

3. 技能伤害计算与范围判定

伤害计算采用经典的"攻击力---防御减免"模型，兼顾物理与法术双属性：

float ad = Define.AD + caster.Attributes.AD * Define.ADFactor;
float ap = Define.AP + caster.Attributes.AP * Define.APFactor;
float addmg = ad * (1 - target.DEF / (target.DEF + 100));
float apdmg = ap * (1 - target.MDEF / (target.MDEF + 100));
float final = (addmg + apdmg) * (IsCrit ? 2 : 1) * Random.Range(0.95f, 1.05f);

其中，防御减免公式 DEF / (DEF + 100) 使得防御收益呈递减曲线，避免数值膨胀。暴击判定基于暴击率属性 CRI 进行随机掷骰。最终伤害值向下取整且至少为 1。

对于范围技能，服务端通过 Battle.FindUnitsInMapRange 方法获取施法点周围指定半径内的所有单位，并对每个目标分别计算伤害。

4. 客户端技能表现同步

客户端收到 SkillCastResponse 后，由 BattleService.OnSkillCast 解析消息，调用对应 Creature 对象的 CastSkill 方法，进而触发 Skill.BeginCast。BeginCast 负责播放施法动画、转向目标、播放吟唱特效等。客户端 Skill.OnUpdate 以 Time.deltaTime 驱动状态机，在 HitTimes 指定的时间点播放命中特效。由于伤害计算已在服务端完成，客户端仅根据 SkillHitResponse 中的 NDamageInfo 更新目标血条和弹出伤害文字，不进行二次计算。

5.3.3 Buff 系统设计

Buff 系统用于实现临时性的属性增益/减益、持续伤害/治疗以及控制效果。本系统中，Buff 的管理与同步遵循以下原则：

- Buff 的添加、移除、伤害跳字均由服务端裁决，客户端仅根据 BuffResponse 更新本地状态和 UI。
- Buff 效果（如眩晕、沉默）通过 BuffEffect 枚举标记，EffectManager 负责维护当前生效的效果集合。

1. Buff 生命周期管理

服务端 BuffManager.AddBuff 在技能命中或特定条件触发时被调用，创建 Buff 对象并加入 Buffs 列表。Buff.OnAdd 方法在 Buff 生效时立即执行效果添加和属性修正，并向 Battle 添加 BuffAction.Add 消息。

属性加成采用叠加式设计：Attributes.Buff.DEF += Attributes.Basic.DEF * Define.DEFRatio。多个 Buff 的属性修正线性叠加，最终属性由 Basic + Buff + Equip 三部分组成。

Buff 的更新在 Buff.Update 中进行：累加持续时间，若配置了伤害间隔则定期触发伤害跳字；持续时间结束时调用 OnRemove 移除 Buff。

2. 客户端 Buff 表现

客户端 BuffManager 同样维护一份本地 Buff 列表。当收到 BuffResponse 时，Creature.DoBUffAction 根据 BuffAction 执行添加或移除操作。添加 Buff 时，客户端在 UI 上显示 Buff 图标及剩余时间；移除时则隐藏图标。对于带有控制效果的 Buff，EffectManager 的 HasEffect 方法可被技能释放逻辑查询，以决定是否允许移动或施法。

5.3.4 飞行物系统实现

对于弓箭、火球等弹道技能，系统采用飞行物（Bullet）对象来模拟投射物运动并延迟伤害判定。服务端和客户端各有一套 Bullet 实现，逻辑对称但职责不同。

1. 服务端飞行物

服务端 Bullet 在技能 DoHit 时创建，根据施法者与目标的距离及弹道速度计算飞行时长：

int distance = skill.Owner.Distance(target);
duration = distance / skill.Define.BulletSpeed;

飞行物采用时间驱动模式，在 Update 中累加飞行时间，到达目标后调用技能的 DoHit 方法执行伤害结算。这种设计避免了服务端每帧更新飞行物位置的开销，适用于大规模战斗场景。

2. 客户端飞行物

客户端 Bullet 与服务端逻辑几乎相同，额外负责驱动飞行物特效的播放。Skill.CastBullet 方法创建 Bullet 对象后，调用 Owner.PlayEffect 播放预设的飞行物特效。特效的实际运动由 FXManager 根据起始点和目标点插值完成，Bullet 仅负责计时并在到期时触发命中表现。

通过客户端与服务端飞行物的并行模拟，玩家在释放弹道技能时能看到流畅的投射物动画，而伤害判定仍由服务端在飞行结束时精确执行，兼顾了表现即时性与逻辑安全性。

5.3.5 战斗消息广播与客户端响应

服务端 Battle.Update 在每帧末尾调用 BroadcastHitsMessage，将本帧产生的 CastSkills、Hits、BuffActions 分别封装为 SkillCastResponse、SkillHitResponse、BuffResponse，并通过 Map.BroadcastBattleResponse 发送给地图内所有玩家。

客户端 BattleService 订阅了这三种响应消息：

- OnSkillCast：驱动目标角色播放施法动画和特效。
- OnSkillHit：解析伤害数据，调用 Creature.DoDamage 更新血量并显示伤害数字。
- OnBuff：更新本地 Buff 列表及 UI。

为了减少网络带宽消耗，消息采用批量发送策略：一帧内的所有战斗事件打包在一个 NetMessageResponse 中下发。此外，只有当前地图中参与战斗或处于视野范围内的玩家才会收到广播（由 Map 的 AOI 模块过滤）。

2. 典型弱网环境（延迟 50-100ms，丢包率 1%）：在此条件下，客户端预测位置与服务端权威位置偏差有所增大，平均偏差在 15-30 逻辑单位之间，偶有超出 50 逻辑单位阈值触发强制修正。修正时画面出现轻微拉扯，但整体体验仍可接受。

3. 极端网络环境（延迟 200-300ms，丢包率 5%）：在此条件下，客户端预测位置与服务端权威位置偏差显著增大，频繁触发阈值修正，画面拉扯感较为明显。战斗技能操作响应延迟增加，但系统状态一致性仍能保持，未出现位置不同步或数据不一致的情况。

1. 系统的各项核心功能均通过测试验证，背包管理、移动同步、技能释放、伤害计算、Buff管理等模块的功能实现完整且符合预期。
2. 在典型网络环境（延迟 ≤ 100ms）下，系统能够提供较为流畅的移动同步与实时战斗体验，位置修正策略有效保证了多端状态的最终一致性。
3. 服务端安全性校验机制可靠，能够有效识别并拒绝非法移动和技能释放请求。
4. 在极端网络环境下，虽然操作手感有所下降，但系统的状态一致性仍然能够得到保证，未出现数据不一致或逻辑错误。

第六章 总结与展望

6.1 论文工作总结

本文围绕"基于状态同步的实时战斗系统设计与实现"这一课题，从网络游戏同步技术的理论基础出发，分析了帧同步与状态同步的适用场景及优劣，针对多人在线角色扮演游戏对安全性、一致性与实时性的综合需求，设计并实现了一套以服务器权威为核心的实时战斗系统原型。

论文主要完成了以下工作：

1. 系统需求分析与架构设计

在深入调研现有网络游戏同步方案的基础上，本文对实时战斗系统进行了全面的需求分析，明确了移动同步、技能释放、伤害计算、Buff管理、背包道具等核心功能需求，以及实时性、安全性、可扩展性等非功能性指标。在此基础上，设计了客户端---服务端分层架构，划分了背包系统、移动同步系统、战斗系统三大功能模块，定义了数据库 E-R 模型与 Protobuf 通信协议。

2. 移动同步模块的实现

基于状态同步原理，实现了完整的移动同步链路。客户端 PlayerInputController 采集玩家输入并执行本地预测移动，通过 MapEntitySyncRequest 上报状态变更；服务端 MapManager 在逻辑帧中校验移动合法性，并通过 MapEntitySyncResponse 向视野内客户端广播实体状态。为平衡响应性与一致性，客户端在 LateUpdate 中实现了基于阈值的位置修正策略，当本地预测与服务端权威位置偏差超过 50 逻辑单位时，执行强制和解。

3. 战斗系统的实现

战斗系统是本文的核心模块。服务端 Battle 对象驱动技能状态机，支持吟唱、持续引导、多段打击等多种技能类型；伤害计算采用"攻击力---防御减免"公式，兼顾物理与法术双属性及暴击判定；Buff 系统通过 EffectManager 管理效果叠加与属性修正，实现了增益、减益、持续伤害等多种效果；飞行物模块采用时间驱动模式，在保证表现流畅性的同时确保伤害判定的服务端权威性。客户端 BattleService 订阅服务端下发的战斗消息，驱动特效播放与 UI 更新。

4. 背包与道具系统的实现

实现了背包、道具、装备等外围功能。以背包系统为例，采用事件驱动的增量同步策略，服务端 StatusManager 将物品变更封装为 StatusNotify 消息推送至客户端，客户端 ItemManager 接收后驱动 UI 局部刷新。数据库方面，背包数据采用二进制 Blob 与行式存储结合的混合策略，兼顾了查询灵活性与读写性能，验证了状态同步机制在非高频数值变更场景下的适用性。

5. 协议设计与网络通信

使用 Protocol Buffer 统一定义了 NetMessage、NEntitySync、SkillCastRequest/Response、StatusNotify 等核心消息结构，保证了客户端与服务端协议的严格一致。网络层基于 TCP 长连接与自定义长度前缀封包格式，实现了可靠的双向通信。

通过上述工作，本文成功构建了一个可运行的实时战斗系统原型，验证了状态同步架构在 MMORPG 场景下的可行性与有效性。

6.2 系统功能特点与技术亮点

经过实现与测试，本系统具备以下功能特点与技术亮点：

1. 服务器权威的状态同步架构

系统严格遵循服务器权威原则，所有涉及数值变更的逻辑（移动、技能伤害、物品交易）均在服务端完成最终裁决。客户端仅作为输入采集器与表现终端，从根本上杜绝了加速、瞬移、属性篡改等客户端作弊行为。

2. 客户端预测与和解机制

为降低网络延迟对操作手感的影响，客户端在收到输入后立即执行本地移动模拟，并在每一帧将本地位置与服务端权威位置进行比对。当偏差超过预设阈值时，触发强制位置修正。该策略在典型网络环境（Ping ≤ 100ms）下，既保证了玩家操作的即时反馈，又维持了多端状态的最终一致性。

3. 事件驱动的增量同步

背包、属性、Buff 等低频状态变更采用 StatusNotify 消息进行增量推送，避免了每次变动都拉取全量数据的带宽浪费。客户端各 UI 模块通过观察者模式订阅数据变更事件，实现局部刷新，提升了界面响应效率。

4. 可扩展的技能与 Buff 系统

技能与 Buff 的数值和行为均由配置表驱动，新增技能或修改数值仅需编辑配置数据，无需改动核心逻辑代码。技能状态机支持吟唱、引导、多段打击、飞行物等多种类型，Buff 系统支持效果叠加、间隔触发、属性修正等丰富机制，具备良好的玩法扩展潜力。

5. 高效的数据持久化策略

背包等高频读写的大数据量模块，采用 Protobuf 序列化后以 varbinary(MAX) 字段存入数据库，将整个背包的读写合并为一次 I/O 操作，显著降低了数据库访问频率。同时保留行式存储表用于关键物品的快速检索，实现了性能与灵活性的平衡。

6.3 存在的不足

尽管本系统完成了预期的设计与实现目标，但在以下方面仍存在不足，有待后续改进：

1. 客户端预测范围有限

当前系统仅在移动模块实现了客户端预测，技能释放、伤害表现等方面仍完全依赖服务端返回结果后再进行表现，导致技能释放存在一定的"点击延迟感"。若能在客户端对技能进行预测施法（即预先播放动画和特效，待服务端确认后再正式结算），可进一步提升操作手感。

2. 位置修正策略较为粗暴

当前的位置和解采用"超过阈值即瞬移"的策略，当网络抖动导致频繁超阈值时，玩家画面可能出现明显的拉扯现象。更优的做法是引入平滑插值或速度场修正，在修正过程中保持视觉连贯性。

3. AOI 管理尚未完善

服务端 Map 类的视野广播逻辑较为简单，目前采用的是遍历全地图玩家的方式。在玩家数量增至百人级别时，广播开销将显著上升。引入十字链表、网格灯塔等经典的 AOI 算法，是提升服务端承载能力的必要优化。

4. 服务端物理模拟缺失

当前移动校验仅检查了速度上限，未对障碍物碰撞、地形高度进行严格的服务端物理模拟。恶意客户端可能通过修改本地碰撞数据穿越墙壁。服务端集成物理引擎或高度图采样，可进一步提升反作弊能力。

5. 战斗平衡性未经充分测试

技能数值、属性成长曲线目前仅基于简单公式设定，未经过大规模玩家测试与数据调优。数值平衡性直接影响游戏的长线生命力，需要持续迭代。

6. 缺少压力测试与性能调优

受限于开发环境，系统未在真实多用户并发场景下进行压力测试，服务端逻辑帧的 CPU 占用、内存分配、网络吞吐量等指标缺乏量化数据支撑。

6.4 未来展望

针对上述不足，结合当前游戏开发技术的发展趋势，本系统可在以下方向进行拓展与优化：

1. 完善客户端预测与回滚机制

参考《守望先锋》《Valorant》等竞技游戏的实践，在技能系统引入客户端预测与回滚（Rollback）机制。客户端在发送技能请求的同时立即播放施法动作，若服务端返回拒绝或修正结果，则回滚至正确状态。该技术可显著降低操作延迟感，是提升竞技体验的关键。

2. 引入高级 AOI 算法

采用十字链表或基于网格的空间索引替代遍历广播，将广播消息的复杂度从 O(n) 降至 O(k)（k 为视野内玩家数），大幅提升服务端在百人同屏场景下的承载能力。

3. 服务端集成确定性物理引擎

在服务端引入轻量级确定性物理引擎（如自研的简单碰撞检测或 BEPUphysics），对玩家移动进行精确的障碍物校验，进一步压缩客户端作弊空间。

4. 战斗系统的深度扩展

- 技能连招系统：支持技能序列检测与组合技触发，丰富战斗策略性。
- 天赋与专精系统：允许玩家自定义技能效果（如增加段数、减少冷却），提升角色 Build 多样性。
- 副本与 BOSS 战：基于现有战斗框架，实现多人组队副本及 BOSS 的复杂 AI 行为树。

5. 运维与监控体系建设

部署独立的日志分析服务与性能监控面板，实时采集服务端各模块的 CPU 耗时、内存分配、GC 频率等指标，为性能调优和容量规划提供数据支撑。

6. 移动端适配

当前客户端基于 Unity 引擎开发，具备跨平台潜力。后续可针对移动设备的触屏操作进行 UI 与交互优化，将系统移植至 Android / iOS 平台，拓宽应用场景。
