## 0. 什么是 Prompt ？

**Prompt（提示词）** 就是你写给 AI 的指令文本。它的本质是用**结构化的自然语言**告诉 AI 三件事：

| 要素 | 说明 | 示例 |
|------|------|------|
| **背景（Context）** | 做什么项目、有什么约束 | "这是一个 Unity UI 开发项目，使用 UniRx 框架" |
| **任务（Task）** | 你希望 AI 完成什么 | "分析这三张配置表的关联关系" |
| **格式（Format）** | 你希望 AI 如何输出 | "输出 C# 数据模型类定义，标注外键字段" |

**类比**：Prompt 之于 AI，就像函数签名 + 参数之于一段代码。模糊的 Prompt（"帮我改改"）≈ `doSomething(data)`；结构化的 Prompt（"对比方案 A 和 B，从性能、可维护性、成本分析"）≈ `compare(A, B, metrics: ["perf", "maint", "cost"])`。

## 1. 通用任务启动模板

> **适用场景**：向 AI 发起任何工程任务的标准化格式

```
【任务目标】一句话描述要完成什么

【涉及文件/模块】
  - 文件A: 路径 + 作用说明
  - 文件B: 路径 + 作用说明

【关键约束】
  - 技术约束（框架、语言、API 限制）
  - 业务约束（不能改的逻辑、数据格式要求）

【期望产出】
  - 具体产出物描述

【参考案例】（可选）
  - 类似实现或设计参考
```

**AI 预期产出**：按你给的约束和产出要求给出方案或代码。

## 2. 策划表分析类

### 2.1 表关系分析

> **适用场景**：分析 3-4 张配置表之间的关联关系

```
【输入】以下三张表 [粘贴表1/表2/表3]
【任务】
1. 画表关系图（主键-外键）
2. 给出数据获取方法设计（输入 → 输出）
3. 明确字段类型和可空性
4. 输出 C# 数据模型 class 定义
```

**AI 预期产出**：表关联说明 + 数据类定义。

### 2.2 关联分析 + 数据获取设计

> **适用场景**：需要分析多表关联，并设计数据获取路径

```
表A: BuildingConfig
| BuildingID | BuildingType | BuildingName | UnlockLevel | FunctionList |
|------------|-------------|--------------|-------------|--------------|
| 1001       | 1           | 主城         | 1           | 1001,1002    |

表B: FunctionConfig
| FunctionID | FunctionName | OpenType | TargetPanelID |
|------------|-------------|----------|---------------|

表C: PanelConfig
| PanelID | PanelName | PrefabPath | NeedCache |

【问题】这三张表之间是什么关联关系？按照 Unity UI 开发的数据获取需求，帮我：
1. 画出表关系图（主键-外键关联）
2. 给出一个数据获取方法的设计（输入：BuildingID → 输出：完整的 UI 展示数据）
3. 考虑性能：一次获取还是懒加载？
```

**AI 预期产出**：表关联图 + 数据获取方法设计 + 性能建议。

### 2.3 数据存取服务骨架生成

> **适用场景**：确认表关系后，生成服务类代码

```
基于上述表关系分析，帮我生成一个 BuildingDataService 的 C# 骨架：
- 方法：GetBuildingDisplayData(int buildingID) → BuildingDisplayData
- 思路：先从 BuildingConfig 取基础信息 → 根据 FunctionList 查 FunctionConfig → 再根据 TargetPanelID 查 PanelConfig
- 只做数据组合，不做 UI 绑定
- 标记 TODO 位置给我补
```

**AI 预期产出**：C# 服务类骨架代码，含 TODO 标记。

## 3. UI 开发类（三层分解法）

> **适用场景**：UI 界面开发，从 UI 元素、数据、交互三层逐步推进

### 3.1 第一层：UI 元素绑定（View）

```
【Prompt】
分析以下功能需求，列出所有需要的 UI 元素，区分通用元素和特定元素，并画出层级结构。

【需在 PLAN 阶段明确的 UI 信息】
  ├─ 列出所有 UI 元素（Button、Text、Image、List 等）
  ├─ 区分通用元素 vs 特定元素
  │   ├─ 通用元素：可抽离复用（关闭按钮、返回按钮、确认弹窗）
  │   └─ 特定元素：本界面独有（建筑交互列表、角色详情面板）
  ├─ 元素命名规范（btn_xxx, txt_xxx, panel_xxx, item_xxx）
  └─ 元素层级关系（父子嵌套结构）
```

**AI 预期产出**：UI 元素清单 + 层级结构图 + 通用/特定分类。

### 3.2 第二层：数据关联（Model）

```
【Prompt】
根据 UI 元素列表，定义每个元素绑定的数据结构，包括数据来源、字段类型和格式化规则。

【需在 PLAN 阶段明确的数据信息】
  ├─ 数据来源（配置表 / 服务器下发 / 本地计算）
  ├─ 数据字段（字段名、类型、取值范围）
  ├─ 数据流转（谁写入、谁读取、何时更新）
  ├─ 显示格式 vs 存储格式（时间、数值千分位、枚举映射）
  ├─ 空值/默认值处理策略
  └─ 数据变更通知机制（事件、回调、脏标记）
```

**AI 预期产出**：数据结构定义 + 字段说明 + 格式化规则。

### 3.3 第三层：交互逻辑（Controller）

```
【Prompt】
基于 UI 元素和数据模型，设计完整的按钮事件绑定方案和页面跳转逻辑。

【需在 PLAN 阶段明确的交互信息】
  ├─ 绑定事件清单
  │   ├─ 按钮点击（核心逻辑 / 跳转 / 关闭）
  │   ├─ 列表项选择 / 长按
  │   ├─ 输入框变更
  │   └─ 拖拽 / 滑动等手势
  ├─ 页面跳转逻辑
  │   ├─ 跳转目标 + 携带参数
  │   ├─ 是否需要返回栈管理
  │   └─ 跳转前后的状态处理（加载、缓存、清理）
  └─ UI 刷新时机
      ├─ 打开界面时（首次初始化）
      ├─ 数据变更时（响应式刷新）
      └─ 从其他界面返回时（恢复状态）
```

**AI 预期产出**：事件绑定方案 + 页面跳转逻辑 + 刷新时机说明。

## 4. 代码审查类

### 4.1 通用代码审查

> **适用场景**：写完代码后让 AI 查问题

```
【任务】审查以下代码
【重点】空指针风险、资源泄漏、性能问题、Unity 生命周期问题
【输出格式】按严重程度排序：[严重] [中等] [建议] 每项标注代码行号
[粘贴代码]
```

**AI 预期产出**：按严重程度排序的审查报告（含行号）。

### 4.2 六象限 Q2 代码审查（增强版）

```
请审查以下代码，从以下维度分析：
1. 空指针/空引用风险
2. 资源泄漏（协程、事件监听、资源加载）
3. 性能问题（频繁 GC、不必要的 Update、重复查找）
4. 与 Unity 生命周期的交互问题
5. 命名和代码规范一致性

按 [严重] [中等] [建议] 三级标记，每项标注行号。
```

**AI 预期产出**：多维度审查报告。

## 5. Bug 定位类

> **适用场景**：崩溃、空引用、UI 数据不刷新

```
【现象】[描述 Bug：什么操作 → 期望什么 → 实际什么]
【报错】[粘贴日志/堆栈]
【相关代码】[粘贴疑似有问题的代码段]
【任务】帮我定位可能的原因，按可能性从高到低排序
```

**AI 预期产出**：按可能性排序的根因分析 + 修复建议。

## 6. 接手旧模块 / 重构类

> **适用场景**：接手别人写的代码，需要快速理解并判断是否重构

### 6.1 Step 1：快速理解（5 min）

```
请阅读以下文件 [粘贴 A.cs 的核心代码]，帮我总结：
1. 这个类/模块的主要职责是什么？
2. 它依赖哪些外部数据或模块？
3. 它对外暴露了哪些方法/事件？
用三点式输出，每点不超过 50 字。
```

**AI 预期产出**：三点式摘要，每点 ≤ 50 字。

### 6.2 Step 2：依赖图谱（5 min）

```
基于上述分析，画出这个模块的依赖关系图：
- 它调用了哪些其他模块？
- 哪些模块调用了它？
- 数据是怎么流进来的？又是怎么出去的？
文本格式输出即可，用 → 符号表示依赖方向。
```

**AI 预期产出**：文本格式依赖图，`A → B → C` 示意。

### 6.3 Step 4：重构建议（5 min）

```
基于上述分析，如果要将这个模块拆分为 [数据层/业务层/展示层/常量] 四部分，请给出具体方案：
1. 新建哪些类？
2. 每个类的职责和关键方法
3. 哪些代码可以复用、哪些需要重写
4. 风险点提示
```

**AI 预期产出**：重构方案（新建类清单 + 职责说明 + 复用/重写清单 + 风险点）。

## 7. 通用 Prompt 模式

> **适用场景**：日常高频任务，按模式复用

### 模式 1：对比分析型

```
对比方案A和方案B，从维护性、性能、实现成本三个维度分析，给出推荐
```

### 模式 2：风险预警型

```
我准备修改 [X模块]，帮我列出所有可能受影响的文件和场景
```

### 模式 3：模板生成型

```
基于以下需求，生成一个 [UI界面/数据类/工具方法] 的完整骨架代码，标记TODO位置
```

### 模式 4：审查型

```
检查以下代码是否有空指针风险、资源泄漏、性能问题，按严重程度排序
```

### 模式 5：文档反查型

```
这是旧代码的改动记录 [粘贴diff]，帮我总结改了什么、为什么改、有无遗留问题
```
