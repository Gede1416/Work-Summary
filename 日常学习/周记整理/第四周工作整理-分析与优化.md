## 2. 策划表 AI 分析实操方案

你的原始笔记中只提了"导入AI分析表关系"的想法，但缺少落地方案。以下是可执行步骤：

### 2.1 给 AI 什么格式的数据？

不要把整张 Excel/CSV 直接丢给 AI（容易被截断、效果差）。推荐格式：

```
表A: BuildingConfig
| BuildingID | BuildingType | BuildingName | UnlockLevel | FunctionList |
|------------|-------------|--------------|-------------|--------------|
| 1001       | 1           | 主城         | 1           | 1001,1002    |
| 1002       | 2           | 铁匠铺       | 5           | 2001,2002    |

表B: FunctionConfig
| FunctionID | FunctionName | OpenType | TargetPanelID |
|------------|-------------|----------|---------------|
| 1001       | 查看详情     | Panel    | 101           |
| 1002       | 升级         | Panel    | 102           |

表C: PanelConfig
| PanelID | PanelName | PrefabPath | NeedCache |
|---------|----------|------------|-----------|
| 101     | 建筑详情   | UI/BuildingDetail | true |
| 102     | 升级界面   | UI/UpgradePanel   | false |

【问题】这三张表之间是什么关联关系？按照 Unity UI 开发的数据获取需求，帮我：
1. 画出表关系图（主键-外键关联）
2. 给出一个数据获取方法的设计（输入：BuildingID → 输出：完整的 UI 展示数据）
3. 考虑性能：一次获取还是懒加载？
```

### 2.2 关键要点

- **截断控制**：一次不要超过 3-4 张关联表，否则 AI 容易遗漏关系
- **明确输出格式**：要求 AI 输出结构体/类定义，而不是文字描述
- **数据流向**：让 AI 画清楚"用户点击 → 查哪几张表 → 组合成什么数据 → 传给哪个 UI 组件"
- **边界标记**：标注哪些字段是索引键、哪些是可空字段、默认值是什么

### 2.3 进阶用法：让 AI 生成数据存取服务

在确认表关系后，可以进一步让 AI 生成服务类骨架：

```
基于上述表关系分析，帮我生成一个 BuildingDataService 的 C# 骨架：
- 方法：GetBuildingDisplayData(int buildingID) → BuildingDisplayData
- 思路：先从 BuildingConfig 取基础信息 → 根据 FunctionList 查 FunctionConfig → 再根据 TargetPanelID 查 PanelConfig
- 只做数据组合，不做 UI 绑定
- 标记 TODO 位置给我补
```

## 3. 程序架构分层详解

你的五层划分（数据展示/缓存/获取/修饰/逻辑判定 + 常量）方向完全正确。以下是在此基础上的完整架构补充：

```
┌─────────────────────────────────────────────────────────┐
│  View Layer（数据展示）                                    │
│  ├─ 通用 UI 组件（抽离复用）                                │
│  ├─ 页面级 UI 控制器（PageController）                      │
│  ├─ UI 元素绑定 + 事件注册                                 │
│  └─ 刷新时机：OnShow / OnDataChanged / OnLanguageChanged    │
├─────────────────────────────────────────────────────────┤
│  ViewModel Layer（数据绑定层）                              │
│  ├─ 为 View 提供格式化后的数据                              │
│  ├─ 实现 INotifyPropertyChanged / Observer 模式            │
│  └─ 处理 UI 交互输入 → 调用 Service                        │
├─────────────────────────────────────────────────────────┤
│  Service Layer（业务逻辑 / 逻辑判定）                       │
│  ├─ 纯 C# 逻辑，不依赖 UnityEngine                         │
│  ├─ 可单元测试                                            │
│  └─ 异常捕获 + 错误码返回                                  │
├─────────────────────────────────────────────────────────┤
│  Data Layer（数据获取 + 缓存）                              │
│  ├─ 数据来源：策划表 / 服务器 / 本地存储                    │
│  │   ├─ TableProvider（表数据读取 + 缓存）                  │
│  │   ├─ NetworkProvider（网络请求 + 重试 + 超时）            │
│  │   └─ LocalProvider（PlayerPrefs / 本地 JSON）            │
│  ├─ 数据修饰（DataDecorator）                              │
│  │   ├─ 格式化（时间、数值、枚举→文本映射）                  │
│  │   ├─ 空值处理（空状态→默认值/占位文案）                   │
│  │   └─ 合并转换（多表关联→展示模型）                       │
│  └─ Observer 注册点：get/set → 追踪数据修改                 │
├─────────────────────────────────────────────────────────┤
│  Common（常量 + 工具 + 配置）                              │
│  ├─ 常量类（Const_xxx）：路径/Key/阈值                     │
│  ├─ 工具类（Utility）：通用格式化、校验、时间处理             │
│  └─ 配置类（AppConfig）：开关、环境、调试模式                 │
└─────────────────────────────────────────────────────────┘
```

### 3.1 数据流转方向

```
[策划表/服务器] ──→ Data Layer（原始数据）
                        │
                        ▼
                   DataDecorator（格式化/合并）
                        │
                        ▼
                   ViewModel（已格式化数据 + Observer）
                        │
                        ▼
                   View（UI 绑定 + 刷新）
```

### 3.2 关键设计决策

| 决策点                            | 推荐做法                  | 原因                                    |
|-----------------------------------|--------------------------|-----------------------------------------|
| View 是否直接访问 Data Layer      | ❌ 不直接                | 绕过 ViewModel 会导致数据格式不一致       |
| Service 是否依赖 MonoBehaviour    | ❌ 不应依赖              | 可测试性 + 脱离 Unity 生命周期            |
| DataDecorator 放哪一层            | 独立静态方法类            | 便于多 ViewModel 复用相同格式化逻辑        |
| Observer 注册时机                 | 打开界面时注册，关闭时注销 | 防止内存泄漏                              |

### 3.3 补充：你笔记中缺失的错误处理层

```
ErrorHandler（错误处理）
  ├─ 网络错误 → 重试弹窗 / 缓存兜底
  ├─ 表数据缺失 → 默认值 + 日志警告
  ├─ 空引用保护 → 防御性检查（?. / ?? / if-null）
  └─ UI 刷新失败 → 回退到上一次有效数据
```

建议在 Service Layer 与 ViewModel 之间插入一个 ErrorHandler 中间件。

## 4. 重复造轮子的 AI 辅助流程

你的原始笔记提到"接手别人做的业务，通过 AI 整理程序架构功能模块"，这个想法很好。以下是可执行的 4 步流程：

### Step 1：AI 快速理解（5 min）

```
Prompt:
"请阅读以下文件 [粘贴 A.cs 的核心代码]，帮我总结：
1. 这个类/模块的主要职责是什么？
2. 它依赖哪些外部数据或模块？
3. 它对外暴露了哪些方法/事件？
用三点式输出，每点不超过 50 字。"
```

### Step 2：AI 输出依赖图谱（5 min）

```
Prompt:
"基于上述分析，画出这个模块的依赖关系图：
- 它调用了哪些其他模块？
- 哪些模块调用了它？
- 数据是怎么流进来的？又是怎么出去的？
文本格式输出即可，用 → 符号表示依赖方向。"
```

### Step 3：对比你的架构，找差异

AI 输出后，你对照你笔记中的五层架构，看看这个模块属于哪一层、是否需要拆分到不同层。如果发现：
- 数据获取和 UI 逻辑混在一起 → 需要抽离 Service
- 硬编码常量散布各处 → 需要收拢到 Const 类
- 格式化逻辑重复 → 需要抽离 DataDecorator

### Step 4：AI 生成重构建议（5 min）

```
Prompt:
"基于上述分析，如果要将这个模块拆分为 [数据层/业务层/展示层/常量] 四部分，请给出具体方案：
1. 新建哪些类？
2. 每个类的职责和关键方法
3. 哪些代码可以复用、哪些需要重写
4. 风险点提示
```

### 预期效果

- 接手旧模块的理解时间：从 2-4 小时 → 20-30 分钟
- 重构方案设计时间：从 1 天 → 1 小时
- 关键风险不遗漏

## 5. 疑问解答：Reactive / Mbox 监听失效

你在原始笔记中提出：**"Reactive Mbox 在有静态数据的时候无法监听数据变化？"**

### 5.1 问题定位

这是一个非常精准的观察。Reactive 系列框架（UniRx / Mbox / R3）在以下场景确实可能不触发：

```
// ❌ 错误场景：静态数据的赋值
public static int PlayerLevel;
PlayerLevel = 5;  // 不会触发任何监听

// ❌ 错误场景：修改引用类型的内部字段
public static ReactiveProperty<int> Level = new();
// 但是外部直改内部的字段，不走 Property
```

### 5.2 解决方案

```
方案 A（推荐）：使用静态 ReactiveProperty + 方法封装
  public static class PlayerData
  {
      private static ReactiveProperty<int> _level = new(1);
      public static IReadOnlyReactiveProperty<int> Level => _level;
    
      public static void SetLevel(int value) => _level.Value = value;
  }
  // 优点：所有修改都走 SetLevel，任何读都通过 IReadOnlyReactiveProperty 订阅

方案 B：手动触发通知
  private static int _level;
  public static int Level
  {
      get => _level;
      set
      {
          _level = value;
          MessageBroker.Default.Publish(new LevelChangedEvent(value));
      }
  }
  // 适合不想引入 ReactiveProperty 但需要通知的场景

方案 C：定位问题
  如果使用第三方框架，先确认：
  □ 是否在 Initialize 时注册了监听（订阅晚了错过通知）
  □ 是否在 OnDisable/OnDestroy 时注销了监听（提前销毁）
  □ 静态数据是否在场景切换时被重置（DontDestroyOnLoad 问题）
```

### 5.3 调试建议

在 ReactiveProperty 的 setter 中添加日志追踪，可以快速定位是"没赋值成功"还是"赋值了但没触发":

```
private static ReactiveProperty<int> _level = new(1)
{
    // 如果需要调试，加一个订阅日志
    // 或在 SetLevel 中添加 Debug.Log
};
public static void SetLevel(int val)
{
    Debug.Log($"[PlayerData] SetLevel: {val}");
    _level.Value = val;
}
```
