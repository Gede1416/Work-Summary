# Plan → Act AI 协作 SOP（标准作业流程）

> 适用工具：Cursor / Cline / Windsurf 等 AI 编程助手  
> 适用场景：Unity/UE UI开发、前端开发、系统重构等工程任务  
> 版本：v1.0 | 日期：2026-04-26

---

## 一、核心协作模型

```
┌─────────────────────────────────────────────────┐
│                  PLAN 模式                        │
│  目标：让 AI 充分理解上下文，产出精确方案          │
│  ├─ 信息收集 → 需求分析 → 方案设计 → 用户确认     │
│  └─ AI 只读文件、搜索代码、提问澄清               │
├─────────────────────────────────────────────────┤
│                  ACT 模式                         │
│  目标：AI 按方案逐步执行，每步可验证               │
│  ├─ 代码生成 → 文件修改 → 命令执行 → 结果验证     │
│  └─ AI 写入文件、执行命令、使用 MCP 工具           │
├─────────────────────────────────────────────────┤
│               Auto-Approve 分级                   │
│  ├─ 安全操作（读文件、安全命令）→ 自动批准         │
│  └─ 危险操作（删除、网络、配置变更）→ 手动确认     │
└─────────────────────────────────────────────────┘
```

## 二、PLAN 阶段 SOP

### 2.1 任务启动模板

向 AI 描述任务时，使用以下结构化模板：

```
【任务目标】一句话描述要完成什么

【涉及文件/模块】
  - 文件A: 路径 + 作用说明
  - 文件B: 路径 + 作用说明

【关键约束】
  - 技术约束（框架、语言、API限制）
  - 业务约束（不能改的逻辑、数据格式要求）

【期望产出】
  - 具体产出物描述

【参考案例】（可选）
  - 类似实现或设计参考
```

### 2.2 UI 开发专项：三层分解法

基于实际 UI 开发工作场景，将任务分解为三层：

#### 第一层：UI 元素绑定（View）

```
PLAN 阶段需要明确：
  ├─ 列出所有 UI 元素（Button、Text、Image、List 等）
  ├─ 区分通用元素 vs 特定元素
  │   ├─ 通用元素：可抽离复用（关闭按钮、返回按钮、确认弹窗）
  │   └─ 特定元素：本界面独有（建筑交互列表、角色详情面板）
  ├─ 元素命名规范（btn_xxx, txt_xxx, panel_xxx, item_xxx）
  └─ 元素层级关系（父子嵌套结构）
```

**AI Prompt 示例：**
> "分析以下功能需求，列出所有需要的 UI 元素，区分通用元素和特定元素，并画出层级结构。"

#### 第二层：数据关联（Model）

```
PLAN 阶段需要明确：
  ├─ 确定 UI 关联的数据结构
  │   ├─ 数据来源（配置表 / 服务器下发 / 本地计算）
  │   ├─ 数据字段（字段名、类型、取值范围）
  │   └─ 数据流转（谁写入、谁读取、何时更新）
  ├─ 关联数据格式化
  │   ├─ 显示格式 vs 存储格式（时间、数值千分位、枚举映射）
  │   └─ 空值/默认值处理策略
  └─ 数据变更通知机制（事件、回调、脏标记）
```

**AI Prompt 示例：**
> "根据 UI 元素列表，定义每个元素绑定的数据结构，包括数据来源、字段类型和格式化规则。"

#### 第三层：交互逻辑（Controller）

```
PLAN 阶段需要明确：
  ├─ 绑定事件清单
  │   ├─ 按钮点击（核心逻辑 / 跳转 / 关闭）
  │   ├─ 列表项选择 / 长按
  │   ├─ 输入框变更
  │   └─ 拖拽 / 滑动等手势
  ├─ 页面跳转逻辑
  │   ├─ 跳转目标 + 携带参数
  │   ├─ 是否需要返回栈管理
  │   └─ 跳转前后的状态处理（加载、缓存、清理）
  └─ UI 刷新时机
      ├─ 打开界面时（首次初始化）
      ├─ 数据变更时（响应式刷新）
      └─ 从其他界面返回时（恢复状态）
```

**AI Prompt 示例：**
> "基于 UI 元素和数据模型，设计完整的按钮事件绑定方案和页面跳转逻辑。"

### 2.3 系统改造专项：变更影响分析

```
PLAN 阶段必须完成的影响分析：

  1. 改动范围界定
     ├─ 直接修改的文件清单
     └─ 间接影响的依赖文件

  2. 接口兼容性检查
     ├─ 对外暴露的 API 是否变化
     ├─ 数据存储格式是否兼容旧版
     └─ 是否需要数据迁移

  3. 回归风险点
     ├─ 哪些旧功能需要重测
     └─ 哪些边界场景容易出问题

  4. 回滚方案
     └─ Git 回滚点建议 + 配置开关方案
```

## 三、ACT 阶段 SOP

### 3.1 执行步骤规范

```
每个 ACT 阶段遵循以下顺序：

Step 1: 文件读取确认
  └─ 让 AI 先读取要修改的文件，确认当前内容

Step 2: 增量修改
  ├─ 优先使用 replace_in_file（精准修改）
  └─ 新建文件使用 write_to_file

Step 3: 每改完一个逻辑单元 → 暂停确认
  └─ 不一次性改多个不相关的文件

Step 4: 修改完成后立即验证
  ├─ 让 AI 重新读取修改后的文件
  └─ 确认改动符合预期
```

### 3.2 UI 开发执行清单

```
□ 创建/修改 UI 元素脚本（绑定引用）
□ 抽取通用元素到公共模块
□ 定义数据结构（Model / ViewModel）
□ 实现数据格式化工具方法
□ 绑定按钮事件（核心交互 + 跳转）
□ 实现 UI 刷新逻辑（Init / OnShow / OnDataChanged）
□ 处理空状态和加载状态
□ 注册/注销事件监听（避免内存泄漏）
□ 添加页面跳转代码（携带参数）
□ 本地自测（打开界面 → 点击按钮 → 跳转 → 返回）
```

### 3.3 资源管理 SOP

```
图片/资源路径变更时：

  ACT 阶段执行：
  ├─ 1. 确认新路径规则（AI 先列出变更前后对照表）
  ├─ 2. 全局搜索旧路径引用（search_files 正则搜索）
  ├─ 3. 逐文件替换（每次替换一个文件，验证后继续）
  ├─ 4. 更新路径常量/配置文件
  └─ 5. 删除废弃资源确认（需手动批准）
```

## 四、工作流优化：通用模块抽离

### 4.1 识别可抽离的通用元素

在 PLAN 阶段，让 AI 自动分析项目中是否存在以下重复模式：

```
可抽离信号：
  ├─ 相同的 UI Prefab/组件在 ≥3 个界面中使用
  ├─ 相同的数据格式化逻辑在多处复制粘贴
  ├─ 相同的按钮事件绑定模式（如"关闭界面"、"返回上一级"）
  ├─ 相同的网络请求错误处理逻辑
  └─ 相同的空状态/加载中/错误状态 UI 展示
```

### 4.2 抽离执行模板

```
当识别到可抽离元素时：

  1. PLAN: AI 输出抽离方案
     ├─ 抽离后的公共模块结构
     ├─ 各调用方的改造点列表
     └─ 兼容性保证措施

  2. ACT: 分步执行
     ├─ Step A: 创建公共模块
     ├─ Step B: 改造一个调用方作为样板
     ├─ Step C: 验证样板正确后批量改造其余调用方
     └─ Step D: 删除旧代码、清理引用
```

## 五、协作技巧与 Prompt 模式

### 5.1 上下文传递技巧

```
✓ 好的做法：
  - 粘贴相关代码片段（关键结构、接口定义）
  - 描述"不要做的事情"比"要做的事情"更有效
  - 明确告知项目的命名约定和代码风格
  - 使用 @file 引用项目中的文件

✗ 避免的做法：
  - 一次性丢入 1000+ 行代码不说明重点
  - 模糊需求："帮我优化一下"（不说优化目标）
  - 在 ACT 模式下频繁要求 AI 重新分析
```

### 5.2 高效 Prompt 模式

```
模式1：对比分析型
  "对比方案A和方案B，从维护性、性能、实现成本三个维度分析，给出推荐"

模式2：风险预警型
  "我准备修改 [X模块]，帮我列出所有可能受影响的文件和场景"

模式3：模板生成型
  "基于以下需求，生成一个 [UI界面/数据类/工具方法] 的完整骨架代码，标记TODO位置"

模式4：审查型
  "检查以下代码是否有空指针风险、资源泄漏、性能问题，按严重程度排序"

模式5：文档反查型
  "这是旧代码的改动记录 [粘贴diff]，帮我总结改了什么、为什么改、有无遗留问题"
```

### 5.3 异常处理 SOP

```
当 AI 输出不符合预期时：

  1. 不要直接否定 → 明确指出哪一部分不对
  2. 提供反例："我要的不是 X，而是类似 Y 的效果"
  3. 要求 AI 先"自检"：让 AI 重新读取修改后的文件，自己发现问题
  4. 缩小范围：如果 AI 一次改动太多导致混乱，拆成更小的子任务
```

## 六、完整示例：建筑交互功能改造

> 对应笔记中的「点击主城建筑弹出交互功能列表 + 添加特定物体功能」

### PLAN 阶段 Prompt

```
【任务目标】点击主城建筑时，弹出交互功能列表，列表中新增"查看详情"和"升级"功能

【涉及文件】
  - MainCityBuilding.cs: 建筑点击处理
  - InteractionPanel.cs: 交互列表 UI（已存在，需改造）
  - BuildingData.cs: 建筑数据模型

【关键约束】
  - 列表需要根据建筑类型动态显示不同功能
  - 特定建筑种类（类型3、类型5）需额外显示"特殊操作"按钮

【请先分析】
  1. 列出需要修改的 UI 元素和数据字段
  2. 画出交互流程图
  3. 给出修改方案，每步说明原因
```

### ACT 阶段执行

```
Step 1: AI 读取 InteractionPanel.cs，确认现有代码结构
Step 2: AI 在 InteractionPanel 中新增按钮绑定代码
Step 3: 确认修改正确后，AI 读取 BuildingData.cs
Step 4: AI 添加建筑类型判断逻辑
Step 5: AI 读取 MainCityBuilding.cs，修改点击回调
Step 6: 全局搜索确认无遗漏引用
Step 7: 输出修改总结（改了什么、测试重点）
```

| 通用模块抽离重构 | 120 min | 15 min (Plan) + 20 min (Act) | ~71% |
