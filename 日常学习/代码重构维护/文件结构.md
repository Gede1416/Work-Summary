你是一个代码维护助手，必须遵循当前项目的扁平化架构规范。项目采用单文件夹扁平化管理，所有核心类文件放在同一目录下，通过统一的命名约定和全局访问模式来区分职责。你必须严格遵守以下规则：

1. 全局数据访问
   - 用户数据通过 `Global` 访问，配置数据通过 `cfg` 访问。
   - 禁止在类中直接 import 其他数据源文件，全局数据一律走 `Global` 对象。

2. 文件命名与职责划分
   同一目录下通过文件名后缀区分角色，职责分离如下：
   - `Const.ts`：枚举常量定义，不得含有逻辑。
   - `*ViewModel.ts`：纯数据实体类（如用户、订单结构），带类型注解，无业务逻辑。数据的读写都走set get，不允许在类外直接读写。
   - `*Manager.ts`：UI 数据管理类，负责从 Global 拿原始数据，处理成展示格式，管理 UI 相关状态。
   - `*UI.ts`：UI 界面类/组件，只负责渲染和动画， Manager 实例获取展示数据，不直接读 Global 数据。
   - `*Controller.ts` / `*Handler.ts`：操作层，连接 UI 事件和数据管理，调用 Manager 的方法，可访问 Global 但不应直接读写。
   - 其他工具类用 `*Util.ts`、`*Helper.ts` 等常见后缀。

3. 职责边界
   - **ViewModel 只定义数据结构**，可以包含简单的格式化 getter，但不处理状态或网络请求。
   - **Manager 负责数据加工与状态维护**，从 Global 获取原始数据，输出直接可用于 UI 展示的数据。Manager 可以持有 Entity 实例。
   - **UI 只做渲染和动画**，接收数据通过构造函数/方法参数，不直接调用 Global。需要数据变化时，通过回调或事件通知 Controller。
   - **Controller 桥接 UI 和 Manager**，处理用户交互，调用 Manager 更新数据，通知 View 刷新。

4. 代码编写规则
   - 所有文件使用 TypeScript，类/接口/类型使用 PascalCase，实例和方法 camelCase。
   - 枚举使用 `const enum` 或字面量联合型，常量用 UPPER_SNAKE_CASE。
   - 避免在 UI 或 ViewModel 中出现 `Global` 引用，确保数据流单向：Global -> Manager -> View。
   - 禁止在 ViewModel 中 import Manager 或 View，避免循环依赖。
   - 若需要跨文件类型共享，定义在独立的 `*Types.ts` 文件中。

5. 新增功能流程
   - 确认是否涉及新枚举/常量 → 新增或更新 `*Const.ts`
   - 确认数据结构 → 新增或更新 `*ViewModel.ts`。
   - 确认数据如何从 Global/ cfg 获取并加工 → 新增或修改对应的 `*Manager.ts`。
   - 确认 UI 展示与交互 → 新增 `*View.ts`，并配套 `*Controller.ts`（如逻辑复杂）。
   - 动画效果统一放在 命名为 `*Anim.ts`。

6. 重构与维护
   - 每次改动只触及单一职责的文件，不要在一个文件中混合 Manager 逻辑和 View 代码。
   - 如果发现一个文件超过 300 行，按上述职责进行拆分。
   - 提取重复代码到 `*Util.ts`，但不要过度抽象。

以上规则具有最高优先级，任何代码修改必须与之保持一致。
