# C++ 调试与性能分析

## 详细描述

C++ 的调试和性能分析直接面向底层，工具链强大但学习曲线相对较陡。

**GDB/LLDB** 是两大调试器：
- **GDB**（GNU Debugger）：Linux 主流选择，命令行驱动，支持断点、单步、变量检查、表达式求值、反向调试。与 GCC 配合使用。
- **LLDB**：Clang 生态的调试器，架构更现代，与 GDB 类似但 API 更清晰。macOS 默认调试器（Xcode 内建）。

**Valgrind** 是 Linux 上内存错误检测的黄金工具。`valgrind --leak-check=full ./myapp` 检测内存泄漏、使用已释放的内存、数组越界访问等。代价是程序运行速度降低 10-50×，适合开发/测试阶段。

**AddressSanitizer (ASan)** 与 Valgrind 功能类似，但通过编译器插桩实现，性能损失更小（约 2×）。GCC 和 Clang 都支持：`-fsanitize=address`。还有 ThreadSanitizer（TSan，数据竞态检测）和 UndefinedBehaviorSanitizer（UBSan，未定义行为检测）。

**perf**（Linux）是系统级性能分析工具，可以采样 CPU 执行、缓存命中/未命中、分支预测、上下文切换等硬件事件。`perf record ./myapp && perf report` 生成火焰图风格的函数时间分布。

**VTune**（Intel）是 Intel 的商业性能分析工具，提供微架构分析（流水线停顿、分支预测失败率）、内存带宽分析、并行效率分析。

**模板实例化爆炸**是 C++ 特有的性能问题——每个模板参数组合生成独立代码，导致编译时间膨胀和二进制膨胀。`nm -C binary | sort` 查看符号大小，`-ftime-report`（GCC）查看模板实例化时间。

## 代码实例

```bash
# ========== GDB 基本工作流 ==========
# 编译（-g 包含调试信息，-O0 禁用优化）
g++ -g -O0 -o myapp myapp.cpp

# 启动 GDB
gdb ./myapp

# GDB 命令：
# (gdb) break main          # 在 main 函数设置断点
# (gdb) run arg1 arg2       # 运行程序
# (gdb) next                # 单步执行（不进入函数）
# (gdb) step                # 单步执行（进入函数）
# (gdb) print variable      # 打印变量值
# (gdb) backtrace           # 显示调用栈
# (gdb) continue            # 继续执行
# (gdb) watch x             # 监视变量变化
# (gdb) info threads        # 查看所有线程
# (gdb) thread 2            # 切换到线程 2
```

```makefile
# ========== 编译时检测 ==========
# AddressSanitizer：内存错误
CXXFLAGS += -fsanitize=address -fno-omit-frame-pointer -g

# UndefinedBehaviorSanitizer：未定义行为
CXXFLAGS += -fsanitize=undefined

# ThreadSanitizer：数据竞态
CXXFLAGS += -fsanitize=thread

# 模板实例化分析（GCC/Clang）
# g++ -ftime-report -ftemplate-depth=1024 ...
```

## 推荐书籍

1. **《The Art of Debugging with GDB, DDD, and Eclipse》** — Norman Matloff / Peter Jay Salzman
   系统讲解 GDB 的全面使用，从基础到多线程调试。

2. **《Systems Performance》（第2版）** — Brendan Gregg
   虽然偏系统层面，但 perf、火焰图的章节是 C++ 性能分析的经典方法。
