# Lua 内存管理

## 详细描述

Lua 从设计之初就将自动内存管理作为核心特性，使用**标记-清除**（Mark-and-Sweep）垃圾回收算法。理解 GC 的工作原理和如何配合它工作，是编写高性能 Lua 代码的关键。

**自动垃圾回收**：Lua 的 GC 自动运行，开发者通常不需要手动干预。当内存达到 GC 阈值时，GC 触发标记阶段（从根对象——全局表、注册表、调用栈——递归标记所有可达对象），然后进入清除阶段（回收未被标记的对象）。Lua 5.1 使用基本的标记-清除，Lua 5.2+ 引入了增量式 GC（可以分步执行，减少暂停时间），Lua 5.4 改进了分代式 GC。

**`__gc` 元方法**：当一个 table 或 full userdata 被 GC 回收时，如果其元表定义了 `__gc` 方法，该方法会被调用。这在需要释放外部资源（文件、网络连接、C 分配的内存）时非常有用。但注意不要依赖 `__gc` 的执行时机——GC 可能在任何时候运行，也可能不运行。

**弱引用表**（Weak Table）是通过 `__mode` 元字段控制的："k" 表示键是弱引用，"v" 表示值是弱引用，"kv" 表示两者都是。弱引用意味着：如果对象仅通过弱表中的引用可达，GC 可以回收它，表中对应的条目会被自动移除。弱引用表在缓存、记忆化（memoization）、对象注册等场景中至关重要。

**避免产生大量临时表**是 Lua 性能优化的首要原则。table 是 Lua 中分配量最大的对象类型——每次用 `{}` 创建一个表字面量，每次函数调用（如果使用了 `...` 并打包为 table），都会产生 GC 压力。使用对象池、复用表、就地修改等策略可以显著减轻 GC 负担。

**强制 GC**：`collectgarbage("collect")` 执行一次完整的 GC 循环。`collectgarbage("count")` 返回当前使用的内存（KB）。在游戏开发中，常在加载屏幕时手动触发 GC 以避免游戏过程中产生卡顿。

**字符串的自动内部化**（interning）：Lua 保证相同内容的字符串存储为同一个对象，因此字符串比较是 O(1) 的引用比较。但这意味着每次创建新字符串（如拼接）都涉及哈希查找，大量字符串操作需要关注性能。

**LuaJIT 的 GC**：LuaJIT 2.1 改进了 GC——默认使用增量式标记-清除，也支持"分代式"选项。LuaJIT 的 FFI 分配的 C 数据不受 GC 管理，需要手动管理。

## 代码实例

```lua
-- ========== 1. GC 监控与控制 ==========
print("Initial memory:", collectgarbage("count"), "KB")

-- 创建大量临时对象
local function createGarbage()
    for i = 1, 10000 do
        local t = { i, i * 2, i * 3, { "nested" } }
    end
end

createGarbage()
print("After creating garbage:", collectgarbage("count"), "KB")

collectgarbage("collect")  -- 强制 GC
print("After GC:", collectgarbage("count"), "KB")

-- 停止 GC（游戏关卡加载期间）
-- collectgarbage("stop")
-- ... 批量创建对象 ...
-- collectgarbage("restart")

-- ========== 2. __gc 元方法 ==========

local Resource = {}
Resource.__index = Resource

function Resource:new(name)
    local r = { name = name }
    setmetatable(r, {
        __gc = function(obj)
            print(string.format("Resource '%s' is being collected", obj.name))
            -- 在这里释放外部资源（C 句柄等）
        end,
        __index = Resource,
    })
    return r
end

-- 使用__gc 作为安全检查
local function useResource()
    local res = Resource:new("temp-file")
    -- 使用 res ...
    -- 函数返回后 res 变成垃圾，GC 某时刻调用 __gc
end

useResource()
collectgarbage("collect")  --> Resource 'temp-file' is being collected

-- ========== 3. 弱引用表 ==========

-- 弱引用值模式：缓存不阻止对象回收
local imageCache = {}
setmetatable(imageCache, { __mode = "v" })

local function loadImage(path)
    local cacheKey = path
    local cached = imageCache[cacheKey]
    if cached then
        print("Cache hit:", path)
        return cached
    end
    
    -- 模拟加载图像（实际可能是大型 C userdata）
    local image = { path = path, pixels = {}, loaded = true }
    imageCache[cacheKey] = image
    print("Cache miss:", path)
    return image
end

local img1 = loadImage("texture.png")
print(imageCache["texture.png"] ~= nil)  --> true

img1 = nil  -- 断开外部引用
collectgarbage("collect")

-- 现在 imageCache 中的条目也被清除了（只有弱引用）
print(imageCache["texture.png"] == nil)  --> true

-- ========== 4. 避免临时表 ==========

-- ❌ 低效：每次循环创建新表
local function badTableUsage(items)
    local result = {}
    for _, item in ipairs(items) do
        local temp = { processed = true, value = item * 2 }
        table.insert(result, temp)
    end
    return result
end

-- ✅ 高效：复用对象池
local function createObjectPool(initialSize)
    local pool = {}
    for i = 1, (initialSize or 10) do
        table.insert(pool, {})
    end
    
    return {
        acquire = function()
            return table.remove(pool) or {}
        end,
        release = function(obj)
            -- 清理对象（但保留表本身）
            for k in pairs(obj) do
                obj[k] = nil
            end
            table.insert(pool, obj)
        end,
    }
end

local pool = createObjectPool(20)

local function betterTableUsage(items)
    local result = {}
    for _, item in ipairs(items) do
        local temp = pool.acquire()
        temp.processed = true
        temp.value = item * 2
        table.insert(result, temp)
    end
    return result
end

-- ========== 5. 字符串与内存 ==========

-- 自动内部化：相同字符串共享
local a = "hello"
local b = "hello"
print(a == b)  --> true（值相等）
-- 内部实现：a 和 b 指向同一个内部化字符串

-- 创建许多临时字符串（注意性能）
local function manyConcatenations(n)
    local result = ""
    for i = 1, n do
        result = result .. tostring(i) .. ","  -- 每次都会创建新字符串
    end
    return result
end

-- ✅ 更好的方式：用 table.concat
local function betterConcat(n)
    local parts = {}
    for i = 1, n do
        parts[i] = tostring(i)
    end
    return table.concat(parts, ",")
end

-- ========== 6. LuaJIT FFI 内存管理 ==========
-- （仅在 LuaJIT 中有效）
--[[
local ffi = require("ffi")

ffi.cdef[[
    void* malloc(size_t size);
    void free(void* ptr);
]]

-- FFI 分配的内存不受 Lua GC 管理！
local ptr = ffi.C.malloc(1024)  -- 需要手动管理
-- 使用 ptr ...
ffi.C.free(ptr)  -- 必须手动释放！
--]]

-- ========== 7. 对象追踪调试 ==========
local trackedObjects = setmetatable({}, { __mode = "k" })

local function track(description)
    local obj = {}
    trackedObjects[obj] = description or "anonymous"
    return obj
end

local a = track("object a")
local b = track("object b")
a = nil  -- 断开引用
collectgarbage("collect")

-- 只能用弱引用键遍历，此时 a 已经被回收
for obj, desc in pairs(trackedObjects) do
    print("Active:", desc)
end
```

## 推荐书籍

1. **《Programming in Lua》（第4版）** — Roberto Ierusalimschy
   第 17 章（Weak Tables and Finalizers）和第 23 章（Garbage Collection 内部）是理解 Lua GC 的权威资料。

2. **《Lua Programming Gems》** — L.H. de Figueiredo
   第 12 章讲解了 GC 调优在实际项目中的应用，包括游戏引擎的内存预算管理。

3. **《LuaJIT》官方文档**（在线）
   LuaJIT 的 GC 实现策略和 `ffi.gc()` 的详细用法是高性能 Lua 应用的必备知识。

4. **《Game Development with Lua》**（多本在线/开源资源）
   游戏场景对 GC 卡顿敏感，这些资源中大量讨论了如何配合 Lua GC 避免帧率波动。
