# Workflow DevKit 异步操作实现 — 完整分析报告

## 一、核心架构概览

Workflow DevKit 的异步模型基于 **事件溯源（Event Sourcing）+ 确定性重放（Deterministic Replay）**，而非传统的 async runtime。整个系统由三个层次构成：

```
┌─────────────────────────────────────────────────────┐
│  用户代码层    async/await + "use step"/"use workflow" │
├─────────────────────────────────────────────────────┤
│  VM 沙箱层     确定性执行环境 (固定时间/种子随机)       │
├─────────────────────────────────────────────────────┤
│  运行时引擎    事件日志重放 + WorkflowSuspension 挂起   │
├─────────────────────────────────────────────────────┤
│  世界层 (World)  事件持久化 + 队列消息 + 流存储         │
└─────────────────────────────────────────────────────┘
```

---

## 二、核心异步机制：WorkflowSuspension 模式

这是整个系统最关键的异步抽象。位于 `packages/core/src/global.ts:46`：

```typescript
class WorkflowSuspension extends Error {
  steps: QueueItem[]   // 所有未决的操作（步骤/钩子/等待）
  stepCount, hookCount, waitCount, ...
}
```

**执行流程：**

1. 工作流函数在 Node.js `vm` 沙箱中确定性执行
2. 用户代码调用 `await stepFn(...)` → 创建一个 Promise（通过 `withResolvers()`）
3. 该 Promise 订阅 `EventsConsumer`，等待事件日志中的匹配事件
4. 如果事件日志中尚无该步骤的完成事件 → 抛出 `WorkflowSuspension` **挂起**整个工作流
5. 运行时捕获该错误，创建 `step_created` 事件，执行步骤（内联或排队），写入 `step_completed` 事件
6. 下次重放时，Promise 从事件日志中找到完成事件，正常 resolve

**关键文件：**
- `packages/core/src/step.ts` — `createUseStep()` 创建步骤 Promise 并订阅事件
- `packages/core/src/workflow.ts:784` — `Promise.race([workflowFn(...), workflowDiscontinuation.promise])`
- `packages/core/src/events-consumer.ts` — 事件顺序消费的状态机

## 三、确定性保证机制

异步操作的确定性由以下机制联合保证：

### 3.1 promiseQueue — 确定性排序

位于 `packages/core/src/workflow.ts:158`，一个 Promise 链：

```typescript
ctx.promiseQueue = Promise.resolve()
// 每次 resolve/reject 都链式调用：
ctx.promiseQueue = ctx.promiseQueue.then(async () => {
  const hydrated = await hydrateStepReturnValue(...);
  resolve(hydrated);
});
```

**作用：** 即使异步反序列化（解密）耗时不同，Promise 也严格按事件日志顺序解析，保证重放一致性。

### 3.2 scheduleWhenIdle — 延迟挂起

位于 `packages/core/src/private.ts:167`，在触发 `WorkflowSuspension` 前轮询 `pendingDeliveries` 计数器直到归零。防止在异步数据传递还在进行时就过早挂起。

### 3.3 VM 沙箱确定性

位于 `packages/core/src/vm/index.ts`：
- `Math.random()` → 种子随机数生成器
- `Date.now()` → 固定的 `startedAt` 时间戳
- `setTimeout`/`setInterval`/`setImmediate` → **全部禁用**（抛出异常）
- `fetch` → 禁用并抛出提示
- 提供自定义 `AbortController`/`Request`/`Response`/`ReadableStream`

### 3.4 SWC 编译层强制

位于 `packages/swc-plugin-workflow/transform/src/lib.rs:59`，Rust 编译插件在编译时强制：
- 标记 `"use step"` 或 `"use workflow"` 的函数**必须**是 `async` 函数
- 在 `"use workflow"` 文件中，只有 async 函数才能被导出

## 四、事件消费系统：EventsConsumer

位于 `packages/core/src/events-consumer.ts`，这是驱动整个异步模型的核心 pub/sub 系统：

```
EventsConsumer
  ├── events: Event[]           // 事件日志
  ├── callbacks: Callback[]     // 订阅者
  └── eventIndex: number        // 当前位置

subscribe(fn) → 注册回调，接收 (event | null)
                返回 Consumed / NotConsumed / Finished
```

各组件订阅的事件：

| 组件 | 订阅事件 | 文件 |
|------|---------|------|
| 步骤 | `step_created`, `step_completed`, `step_failed`, `step_retrying` | `step.ts` |
| 睡眠 | `wait_created`, `wait_completed` | `workflow/sleep.ts` |
| 钩子 | `hook_created`, `hook_received`, `hook_conflict`, `hook_disposed` | `workflow/hook.ts` |
| 取消 | `hook_created`, `hook_received`, `hook_disposed` | `workflow/abort-controller.ts` |

事件通过 `process.nextTick()` 消费，保证确定性顺序。100ms 的延迟检查（`DEFERRED_CHECK_DELAY_MS`）防止异步反序列化时的误报。

## 五、两层执行架构：V1 vs V2

### V1 — 异步步骤处理程序
位于 `packages/core/src/runtime/step-handler.ts`，每个步骤通过独立队列 `__wkf_step_<name>` 异步执行，完成后再排队工作流继续。

### V2 — 组合处理程序（内联优化）
位于 `packages/core/src/runtime.ts:593`，主循环：

```
while(true) {
  1. 加载事件 → 完成过期等待 → runWorkflow() 重放
  2. 若完成 → run_completed，返回
  3. 若 WorkflowSuspension → handleSuspension()
     - 单个步骤且拥有所有权 → 内联执行（避免队列往返）
     - 多个步骤/钩子 → 排队其他，内联一个
     - 等待(sleep) → 返回 timeoutSeconds，让队列驱动重试
  4. 若超时(120s) → 重新排队，不失败
}
```

**所有权机制：** 只有成功写入 `step_created` 事件的处理程序才能执行该步骤（`suspension-handler.ts:50-54`）。这保证了即使多个处理程序竞态进入同一批处理边界，每个步骤也只有**一个所有者**。

## 六、并发控制策略

该项目**不使用**传统的信号量/互斥锁/速率限制器。并发通过以下架构手段管理：

| 机制 | 实现位置 | 原理 |
|------|---------|------|
| **队列驱动** | `runtime/helpers.ts:438` | 工作和步骤通过 `__wkf_workflow_*` / `__wkf_step_*` 队列分发 |
| **事件原子性** | `suspension-handler.ts` | `step_created` 事件的原子写入保证步骤所有权唯一 |
| **批量完成协调** | `runtime.ts:396-424` | 最后完成的处理程序触发内联重放 |
| **ThrottleError** | 世界层（后端） | 429 速率限制，返回 `timeoutSeconds` 延迟重试 |
| **并行操作** | 多处 | `Promise.all()` 用于并行事件创建，`Promise.allSettled()` 用于部分失败容忍 |

### 构建队列（防抖）

位于 `packages/builders/src/build-queue.ts`，一个简洁的 Promise 链式序列化器：

```typescript
function createBuildQueue() {
  let queue = Promise.resolve();
  return (task) => {
    queue = queue.then(task).catch(console.error);
    return queue;
  };
}
```

用于 NestJS、SvelteKit、Vite 等框架集成中的 HMR 防抖。

## 七、取消、重试与超时

### 取消 — 基于钩子的 AbortController

位于 `packages/core/src/workflow/abort-controller.ts`，取消操作被实现为**系统钩子**，以兼容事件溯源：

```
abort() 调用
  → 同步设置 signal.aborted = true
  → 标记钩子为 abortRequested
  → handleSuspension 创建 hook_received 事件 + 流数据包
  → 重放时 EventsConsumer 调用 signal._setAborted(reason)
  → 工作流代码中 abort() 变成无操作（确定性重放）
```

支持 `AbortSignal.any()` 组合多个信号（先到者获胜）。

### 步骤重试

位于 `packages/core/src/runtime/step-executor.ts:130`：
- 默认最大重试次数：`DEFAULT_STEP_MAX_RETRIES = 3`
- 支持 `RetryableError` 自定义 `retryAfter`（毫秒/Duration/Date）
- `FatalError` 跳过重试，直接冒泡到工作流
- 每次重试创建 `step_retrying` 事件，返回 `{ type: 'retry', timeoutSeconds }`

### 多层超时

| 超时类型 | 值 | 位置 | 行为 |
|---------|---|------|------|
| V2 内联循环超时 | 120s | `runtime.ts:119` | 重新排队，不失败 |
| 重放超时 | 240s | `runtime.ts:222` | 3次重试后 `process.exit(1)` |
| 队列交付上限 | 48次 | `constants.ts:13` | 超出后创建失败事件 |

## 八、流架构

使用 Web Streams API 实现大数据传输：

`packages/core/src/flushable-stream.ts` — 自定义 `flushablePipe()`：
- 与 `pipeTo()` 不同，在源流完成**或**用户释放锁且所有写入已刷新时 resolve
- 使用 `setInterval(10ms)` 轮询写入器锁释放
- 跟踪 `pendingOps` 计数确保完整性

`packages/core/src/step/writable-stream.ts` — 步骤关联的可写流：
- `WorkflowServerWritableStream`（服务端写入）→ `SerializeStream`（序列化）→ `flushablePipe`
- `state.promise` 加入 `ctx.ops`，步骤执行器等待其完成

`packages/world-postgres/src/streamer.ts` — PostgreSQL 流持久化：
- 使用 `pg_notify()`/`LISTEN` 进行跨进程实时通知
- 基于游标的分页读取
- `Rc` 引用计数互斥锁控制每流并发

## 总结

Workflow DevKit 没有使用传统的异步运行时（如 tokio、自定义调度器），而是通过 **事件溯源 + 确定性重放** 构建了一个独特的异步模型：

1. **挂起机制** — `WorkflowSuspension` 错误作为用户代码与运行时之间的异步边界
2. **确定性恢复** — `promiseQueue` + VM 沙箱 + 事件日志保证重放一致性
3. **并发安全** — 原子事件创建 + 队列所有权代替传统锁
4. **取消语义** — 通过钩子系统实现可重放的 `AbortController`
5. **内联优化** — V2 运行时让单步骤场景省略队列往返
