# Workflow DevKit 事件溯源与确定性重放机制

## 一、核心问题

Workflow DevKit 需要解决一个关键问题：**长时间运行的工作流如何在服务器无状态环境下可靠执行？**

传统方案（如 Temporal、AWS Step Functions）依赖有状态的后端持续追踪执行进度。Workflow DevKit 选择了不同的路径：**将工作流状态完全外化到事件日志中，通过确定性重放恢复状态**。

```
┌──────────────────────────────────────────────────────────────┐
│                      核心思想                                │
│                                                              │
│  工作流状态 = 重放(事件日志, 工作流代码)                       │
│                                                              │
│  事件日志 是唯一的真相来源 (Single Source of Truth)           │
│  工作流代码 是纯函数: (输入, 事件日志) → 输出                  │
│  VM 沙箱 保证每次重放产生完全相同的效果                        │
└──────────────────────────────────────────────────────────────┘
```

## 二、事件日志：真相的唯一来源

### 2.1 关键事件类型

### 2.2 事件加载

事件通过分页方式从 World 层加载（`runtime/helpers.ts:320-384`）：

```typescript
// runtime/helpers.ts
export async function loadWorkflowRunEvents(
  runId: string,
  afterCursor?: string      // 增量加载的游标
): Promise<{ events: Event[]; cursor: string | null }> {
  const loadedEvents: Event[] = [];
  let cursor: string | null = afterCursor ?? null;
  let hasMore = true;

  while (hasMore) {
    const response = await world.events.list({
      runId,
      pagination: {
        sortOrder: 'asc',       // 升序，保证确定性顺序
        cursor: cursor ?? undefined,
      },
    });
    loadedEvents.push(...response.data);
    hasMore = response.hasMore;
    cursor = response.cursor;
  }

  return { events: loadedEvents, cursor };
}
```

**关键设计决策：**
- 按升序加载（`sortOrder: 'asc'`），确保重放顺序一致
- 支持游标增量加载，避免每次循环重新加载全部事件
- 事件是**不可变的**——一旦写入就不可修改

### 3.2 事件增量加载策略

位于 `runtime.ts:627-692`，采用三级缓存策略：

```
迭代1: cachedEvents = null
  ├── 有预加载事件(来自run_started响应) → 使用预加载事件
  └── 无预加载 → 全量加载 + 获取游标

迭代2+: cachedEvents ≠ null
  ├── 有游标 eventsCursor → 增量加载(仅获取游标之后的新事件)
  │   └── 按 eventId 去重后追加到 cachedEvents
  └── 无游标(仅在使用预加载时) → 全量加载获取游标
```

```typescript
// runtime.ts — 关键代码片段
if (cachedEvents === null) {
  // 首次迭代
  if (preloadedEvents) {
    events = preloadedEvents;
  } else {
    const loaded = await loadWorkflowRunEvents(runId);
    events = loaded.events;
    eventsCursor = loaded.cursor;
  }
} else if (eventsCursor) {
  // 增量加载：仅获取游标之后的新事件
  const loaded = await loadWorkflowRunEvents(runId, eventsCursor);
  if (loaded.events.length > 0) {
    const existingIds = new Set(cachedEvents.map((e) => e.eventId));
    for (const e of loaded.events) {
      if (!existingIds.has(e.eventId)) {
        cachedEvents.push(e);
      }
    }
  }
  eventsCursor = loaded.cursor ?? eventsCursor;
  events = cachedEvents;
}
```

### 3.3 等待完成与并发检测

每次迭代中，运行时检查两个关键条件：

**等待完成** (`runtime.ts:723-764`)：找出所有 `resumeAt` 已到期但还没有 `wait_completed` 事件的 `wait_created`，逐个创建 `wait_completed` 事件。

**并发终止检测** (`runtime.ts:705-720`)：扫描事件日志是否有 `run_completed`/`run_failed`/`run_cancelled`。如果其他处理程序已经终止了运行，当前处理程序直接退出。

### 4.2 挂起触发时机

**由 scheduleWhenIdle 控制**（`private.ts:167`）：

```typescript
// private.ts
export function scheduleWhenIdle(
  ctx: WorkflowOrchestratorContext,
  fn: () => void
): void {
  const check = () => {
    if (ctx.pendingDeliveries > 0) {
      // 仍有数据在传递中 → 等待 promiseQueue 排空后重试
      ctx.promiseQueue.then(() => {
        setTimeout(check, 0);
      });
    } else {
      fn();  // 所有数据传递完成 → 触发挂起
    }
  };
  setTimeout(check, 0);
}
```

**为什么需要 `pendingDeliveries` 计数器？**

当事件重放时，多个 `step_completed`/`hook_received` 的回调可能正在并发地通过 `promiseQueue` 进行异步反序列化（解密）。如果在这些异步操作完成之前就抛出 `WorkflowSuspension`，部分步骤的结果会丢失。

`scheduleWhenIdle` 确保只在所有数据交付完成后才触发挂起。

## 五、EventsConsumer：事件订阅/消费系统

### 5.1 核心设计

位于 `events-consumer.ts`，是驱动所有异步原语的事件分发器：

```typescript
// events-consumer.ts
export class EventsConsumer {
  eventIndex: number;
  readonly events: Event[] = [];
  readonly callbacks: EventConsumerCallback[] = [];

  subscribe(fn: EventConsumerCallback) {
    this.callbacks.push(fn);
    // 取消任何待处理的未消费检查（新回调可能消费事件）
    if (this.pendingUnconsumedCheck !== null) {
      this.unconsumedCheckVersion++;
      this.pendingUnconsumedCheck = null;
      if (this.pendingUnconsumedTimeout !== null) {
        clearTimeout(this.pendingUnconsumedTimeout);
        this.pendingUnconsumedTimeout = null;
      }
    }
    process.nextTick(this.consume);   // 微任务调度
  }

  private consume = () => {
    const currentEvent = this.events[this.eventIndex] ?? null;
    for (let i = 0; i < this.callbacks.length; i++) {
      const callback = this.callbacks[i];
      let handled = EventConsumerResult.NotConsumed;

      handled = callback(currentEvent);

      if (handled === EventConsumerResult.Consumed ||
          handled === EventConsumerResult.Finished) {
        this.eventIndex++;  // 前进到下一个事件
        if (handled === EventConsumerResult.Finished) {
          this.callbacks.splice(i, 1);  // 移除已完成的回调
        }
        process.nextTick(this.consume);  // 继续处理下一个事件
        return;
      }
    }

    // 全部回调返回 NotConsumed → 如果事件非 null，
    // 安排延迟检查（等待 promiseQueue 排空后检测孤立事件）
    if (currentEvent !== null) {
      this.scheduleUnconsumedCheck(currentEvent);
    }
  };
}
```

### 5.2 订阅者注册时机

每种异步原语在工作流代码被调用时注册到 EventsConsumer：

| 原语 | 触发代码 | 订阅事件 | 文件 |
|------|---------|---------|------|
| `await step(x)` | `ctx.eventsConsumer.subscribe(...)` | step_created, step_completed, step_failed, step_retrying | `step.ts:53` |
| `await sleep(ms)` | `ctx.eventsConsumer.subscribe(...)` | wait_created, wait_completed | `workflow/sleep.ts:29` |
| `await hook` | `ctx.eventsConsumer.subscribe(...)` | hook_created, hook_received, hook_conflict, hook_disposed | `workflow/hook.ts:47` |
| `new AbortController()` | `ctx.eventsConsumer.subscribe(...)` | hook_created, hook_received, hook_disposed | `workflow/abort-controller.ts:129` |
| 时间戳更新 | `ctx.eventsConsumer.subscribe(...)` | 所有事件（被动） | `workflow.ts:193` |
| 生命周期 | `ctx.eventsConsumer.subscribe(...)` | run_created, run_started | `workflow.ts:205` |

### 5.3 消费流程示例：Step 的完整生命周期

```
用户代码: await stepFn(arg1, arg2)

1. createUseStep 创建 Promise (withResolvers)
2. 生成 correlationId = "step_<ulid>"
3. 将 StepInvocationQueueItem 放入 ctx.invocationsQueue
4. 注册回调到 EventsConsumer
5. EventsConsumer.consume() 被 process.nextTick 触发

   ┌─ 事件 #N: step_created(correlationId)
   │   回调: 标记 hasCreatedEvent = true, 返回 Consumed
   │
   ├─ 事件 #N+1: step_started(correlationId)
   │   回调: 返回 Consumed (等待后续完成/失败)
   │
   ├─ 事件 #N+2: step_completed(correlationId, result)
   │   回调: pendingDeliveries++
   │         promiseQueue = promiseQueue.then(async () => {
   │           hydrated = await hydrateStepReturnValue(result)
   │           resolve(hydrated)
   │           pendingDeliveries--
   │         })
   │         返回 Finished → 从 callbacks 移除此回调
   │
   └─ 事件结束 (null 事件)
       如果 invocationsQueue 中仍有未决项:
         scheduleWhenIdle(() => {
           onWorkflowError(new WorkflowSuspension(...))
         })
```

## 六、promiseQueue：确定性排序

### 6.1 问题

事件日志保证了事件在存储层面的顺序。但在重放时，异步反序列化操作（解密、反序列化）可能以不确定的顺序完成。如果不加控制：

```
事件日志顺序: stepA_completed → stepB_completed
实际解析顺序: stepB 先解密完成 → stepB result 先 resolve
             → stepA 后解密完成 → stepA result 后 resolve
结果: stepA 的 Promise 在 stepB 之后才 resolve → 不确定!
```

### 6.2 解决方案

位于 `workflow.ts:158`，所有 Promise resolve/reject 都链式追加到 `promiseQueue`：

```typescript
// workflow.ts
const promiseQueueHolder = { current: Promise.resolve() };

// 在 WorkflowOrchestratorContext 中:
get promiseQueue() { return promiseQueueHolder.current; },
set promiseQueue(value) { promiseQueueHolder.current = value; },

// step.ts — 步骤完成时
ctx.pendingDeliveries++;
ctx.promiseQueue = ctx.promiseQueue.then(async () => {
  try {
    const hydratedResult = await hydrateStepReturnValue(
      event.eventData.result,
      ctx.runId, ctx.encryptionKey, ctx.globalThis
    );
    resolve(hydratedResult as Result);
  } catch (error) {
    reject(error);
  } finally {
    ctx.pendingDeliveries--;
  }
});
```

**效果**：即使 stepB 的反序列化比 stepA 快，stepB 的 resolve 也必须等待 stepA 的 promiseQueue.then() 先完成。

### 6.3 pendingDeliveries 计数

`pendingDeliveries` 在反序列化开始时递增，在 finally 块中递减。`scheduleWhenIdle` 轮询此计数器以确保所有数据传递完成后再触发 `WorkflowSuspension`。

## 七、VM 沙箱：确定性执行环境

### 7.1 核心保证

位于 `vm/index.ts:20-125`，VM 沙箱消除所有非确定性来源：

```typescript
// vm/index.ts
export function createContext(options: CreateContextOptions) {
  const { seed, fixedTimestamp } = options;
  const rng = seedrandom(seed);           // 种子随机数
  const context = vmCreateContext();
  const g = runInContext('globalThis', context);

  // 1. Math.random → 种子 PRNG
  g.Math.random = rng;

  // 2. Date → 固定时间戳 (随事件推进)
  g.Date = function Date(...args) {
    if (args.length === 0) return new Date_(fixedTimestamp);
    return new Date_(...args);
  };
  g.Date.now = () => fixedTimestamp;

  // 3. crypto.getRandomValues → 种子 PRNG
  g.crypto = new Proxy(originalCrypto, {
    get(target, prop) {
      if (prop === 'getRandomValues') return getRandomValues;  // 使用种子 RNG
      if (prop === 'randomUUID') return randomUUID;            // 使用种子 RNG
      // ... subtle.digest 放行, generateKey 报错
    }
  });
}
```

## 九、handleSuspension：将挂起状态转换为事件

位于 `suspension-handler.ts:66`，这是挂起状态到事件日志的转换器：

```typescript
// suspension-handler.ts
export async function handleSuspension({
  suspension, world, run, span, requestId
}): Promise<SuspensionHandlerResult> {
  const runId = run.runId;

  // 按类型分类
  const stepItems = suspension.steps.filter(item => item.type === 'step');
  const allHookItems = suspension.steps.filter(item => item.type === 'hook');
  const waitItems = suspension.steps.filter(item => item.type === 'wait');

  // 分类钩子操作
  const hooksNeedingCreation = allHookItems.filter(item => !item.hasCreatedEvent);
  const hooksNeedingDisposal = allHookItems.filter(item => item.disposed);
  const hooksNeedingAbort = allHookItems.filter(
    item => item.abortRequested && !item.disposed
  );
  const stepsNeedingCreation = stepItems.filter(item => !item.hasCreatedEvent);
  const waitsNeedingCreation = waitItems.filter(item => !item.hasCreatedEvent);

  // 1. 并行创建 hook_created 事件 (先处理钩子, 避免与 webhook 接收者竞争)
  await Promise.all(
    hooksNeedingCreation.map(item =>
      world.events.create(runId, {
        eventType: 'hook_created',
        correlationId: item.correlationId,
        eventData: { token: item.token, metadata, isWebhook, isSystem }
      })
    )
  );

  // 2. 创建 hook_disposed 事件
  // 3. 处理中止请求 (创建 hook_received + 流数据包)

  // 4-5. 并行创建 step_created 和 wait_created
  const createdStepCorrelationIds = new Set<string>();
  const ops: Promise<void>[] = [];

  for (const item of stepItems) {
    if (stepsNeedingCreation.has(item.correlationId)) {
      ops.push((async () => {
        await world.events.create(runId, {
          eventType: 'step_created',
          correlationId: item.correlationId,
          eventData: { stepName, input: dehydratedInput }
        });
        createdStepCorrelationIds.add(item.correlationId);  // 记录所有权
      })());
    }
  }

  await Promise.all(ops);

  // 计算等待的最小超时时间
  const minTimeoutSeconds = /* 最近到期等待的秒数 */;

  return {
    pendingSteps: stepItems,
    createdStepCorrelationIds,   // 所有权信息
    timeoutSeconds: minTimeoutSeconds,
    hasHookConflict,
  };
}
```

### 9.1 步骤所有权设计

**关键设计**：只有成功写入 `step_created` 事件的处理程序才能执行该步骤。

```typescript
// runtime.ts:877
const ownedPendingSteps = pendingSteps.filter(s =>
  suspensionResult.createdStepCorrelationIds.has(s.correlationId)
);
```

这保证了即使多个处理程序并竞进入同一批处理边界，每个步骤也只有一个所有者。`step_created` 的原子性由 World 层保证（`EntityConflictError` 处理重复创建）。

### 9.2 有等待时的内联退避

```typescript
// runtime.ts:899
const inlineStep =
  suspensionResult.timeoutSeconds === undefined
    ? ownedPendingSteps[0]
    : undefined;   // 有待处理等待 → 跳过内联执行
```

**原因**：内联 `executeStep()` 会阻塞整个处理程序。如果同时有 `sleep(1s)` 和耗时 60s 的步骤，内联执行会让等待计时器无法按时触发——步骤完成前 `wait_completed` 永远不会被创建，`Promise.race(step, sleep)` 的语义被破坏。

## 十、AbortController：基于钩子的取消

### 10.1 为什么取消需要特殊处理？

如果在工作流 VM 中直接使用普通的 `AbortController`：
- 重放时无法确定地重现 `abort()` 的调用时机
- 取消信号无法传播到其他计算实例上的步骤

Workflow DevKit 将 `AbortController` 实现为**系统钩子**。

### 10.2 实现细节

位于 `workflow/abort-controller.ts:102`：

```
首次运行:
  1. new AbortController()
     └── 在 invocationsQueue 中创建系统钩子 (isSystem: true)
     └── 订阅 EventsConsumer 等待 hook_received 事件

  2. controller.abort(reason)
     ├── signal._setAborted(reason)  ← 同步设置状态
     ├── 标记钩子: abortRequested = true
     └── handleSuspension 处理时:
         ├── 创建 hook_received 事件 (携带脱水的中止原因)
         └── 写入流数据包 (实时传播到运行中的步骤)

重放时:
  1. 事件消费者遇到 hook_received 事件
     └── 通过 promiseQueue 异步水化原因
     └── signal._setAborted(reason)  ← 从事件日志恢复状态
     └── 移除队列项

  2. controller.abort() 被调用
     └── if (this.signal.aborted) return;  ← 无操作 (已经中止)
```

## 十三、总结

Workflow DevKit 通过 **事件溯源 + 确定性重放** 实现了一个无需有状态后端的持久工作流引擎：

1. **事件日志是唯一真相来源** — 工作流状态不存储在内存，而是通过重放事件日志恢复
2. **WorkflowSuspension 是异步边界** — 当一个步骤的结果尚未出现在事件日志中时，抛出此错误暂停工作流；事件到达后重放恢复
3. **EventsConsumer 是异步协调核心** — 所有异步原语（step/hook/sleep/abort）都通过订阅事件日志来驱动 Promise 的 resolve/reject
4. **promiseQueue 保证确定性** — 即使异步反序列化耗时不同，Promise 也严格按事件日志顺序解析
5. **VM 沙箱消除非确定性** — 固定时间、种子随机、禁止异步调度，确保每次重放产生相同结果
6. **V2 内联优化减少延迟** — 单步骤场景下省略队列往返，通过原子事件创建保证步骤所有权
