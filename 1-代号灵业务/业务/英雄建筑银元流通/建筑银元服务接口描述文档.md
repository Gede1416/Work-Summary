# 建筑银元服务接口描述文档

> 银元 itemId: `300500`（`E_ItemID.Gold`），本文档描述客户端与服务端之间涉及银元变动（消耗/产出）的所有网络协议接口。

---

## 一、银元消耗接口

### 1.1 建筑升级 — 扣除玩家银元

**协议名**：`McsUpgradeBuildingReq` / `MscUpgradeBuildingRsp`

**调用方法**：`Network.Ins.helper.UpgradeBuildingReqAsync()`

**触发条件**：玩家点击建筑升级按钮，且建筑配置 `CostIdList` 中包含 `300500`（银元）

**请求体** `CSUpgradeBuildingReq`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `BuildingData.BuildingId` | uint | 当前建筑配置ID |
| `BuildingData.BuildingType` | uint | 建筑类型（102~112） |

**响应体** `SCUpgradeBuildingRsp`：服务端验证并扣除材料（含银元），返回升级结果

**涉及建筑**：全部建筑类型（102/103/107/108/109/110/111/112）

**调用链路**：
```
JianzhuUILevelupController.OnClickLevelUp()
  → BuildingSystem.RequestUpgradeBuilding(buildingId, callback)
    [BuildingSystem_Net_Base.cs:94]
  → helper.UpgradeBuildingReqAsync(CSUpgradeBuildingReq)
    [ClientNetSender.cs:1315]
```

**银元扣除逻辑**：由服务端根据 `building` 配置表的 `CostIdList/CostCountList` 执行扣除。客户端仅做前置检查（`IsItemEnough`），不直接操作银元。

**响应体** `SCForgeHeroEquipmentRsp`：服务端扣除材料（含银元），产出装备

**涉及建筑**：103 裁缝机、107 锻造台、108 符儡仪

**调用链路**：

**银元扣除逻辑**：由服务端解析 `hero_equipment` 表的 `ForgeCostList`（格式 `"itemId|count"`），当 itemId=300500 时扣除对应 count 的银元。

### 1.6 交易所自动交易 — 扣除玩家银元

**协议名**：`McsHeroOrderTradeReq` / `MscHeroOrderTradeRsp`

**调用方法**：`Network.Ins.helper.HeroOrderTradeReqAsync()`

**触发条件**：英雄走到交易所（104鬼眼秤），触发自动交易

**请求体** `CSHeroOrderTradeReq`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `HeroUuid` | ulong | 英雄UUID |
| `ItemIdList` | RepeatedField\<uint\> | 要交易的物品ID列表 |

**响应体** `SCHeroOrderTradeRsp`：服务端扣除玩家银元，英雄获得物品

**客户端预计算逻辑**（`OrderSystem_Util.cs:90`）：
```
goldCount = ItemSystem.Ins.GetItemCount(E_ItemID.Gold)
for each item in hero bag:
    price = item.Param1List[0]
    exchangeCount = min(item.count, goldCount / price)
    goldCount -= exchangeCount * price
```
客户端计算出理论可交易量后发送请求，服务端实际扣除并结算。

**调用链路**：
```
OrderSystem.AutoTrade(heroController)
  [OrderSystem_Util.cs:90]
  → 计算可交易列表 → CSHeroOrderTradeReq
  → helper.HeroOrderTradeReqAsync(req)
    [ClientNetSender.cs:1828]
```

## 三、服务端推送（Sync）消息

以下为服务端主动下发的消息，与银元数据同步相关：

| 消息 | 协议名 | 说明 |
|------|--------|------|
| 初始化建筑数据 | `MscInitBuildingInfo` | 登录时下发所有建筑数据（可能含银元余额） |
| 同步建筑数据 | `MscSyncBuildingData` | 建筑升级后数据变更同步 |
| 初始化恢复信息 | `MscInitRecoverInfo` | 登录时下发英雄恢复队列状态 |
| 血量恢复完成 | `MscSyncHeroRecoverHpDone` | HP恢复完成，服务端结算银元 |
| 心情恢复完成 | `MscSyncHeroRecoverMoodDone` | 心情恢复完成，服务端结算银元 |
| 精力恢复完成 | `MscSyncHeroRecoverEnergyDone` | 精力恢复完成，服务端结算银元 |
| 体力恢复完成 | `MscSyncHeroRecoverStrengthDone` | 体力恢复完成，服务端结算银元 |
| 初始化装备数据 | `MscInitEquipmentInfo` | 登录时下发装备数据 |
| 同步装备数据 | `MscSyncEquipmentData` | 装备增/改/删同步 |
| 初始化订单数据 | `MscInitOrderInfo` | 登录时下发交易所订单数据 |
| 同步订单数据 | `MscSyncOrderData` | 订单增/改/删同步 |

> **关于银元产出结算**：英雄使用恢复建筑后，客户端等待 `MscSyncHeroRecoverXxxDone`。服务端在推送此消息前已完成银元结算（根据 `商品售价 = item.Param1List[0] × (10000 + 售价加成万分比) / 10000` 计算并增加玩家银元），但客户端代码中看不到直接操作银元的逻辑 —— 银元数量通过通用背包同步机制更新。

## 四、银元售价计算（客户端定价）

**协议端**：此计算在客户端执行，未通过独立协议。实际结算由服务端执行。

**计算方法**：`RecoverSystem_Util.cs:178` — `GetProductPriceFinal()`
```
最终售价 = item.Param1List[0] × (10000 + 售价加成万分比) / 10000
```

**各建筑售价加成属性**：

| 建筑 | 加成属性ID | 说明 |
|------|-----------|------|
| 102 参王汤 | Attr 230001 | 血量恢复商品售价加成 |
| 109 孟茶摊 | Attr 230011 | 心情恢复商品售价加成 |
| 110 纸扎庐 | Attr 230015 | 精力恢复商品售价加成 |
| 111 鬼火锅 | Attr 230019 | 体力恢复商品售价加成 |

> 上述售价加成来自属性系统（`AttrSystem.Ins.GetAttribute(attrId)`），受建筑等级和玩家养成影响。
