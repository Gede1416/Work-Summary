# 建筑限制英雄升级最大值计算流程

## 1. 概述

渡劫台(E_116) / 白玉床(E_114) 通过**建筑等级**限制英雄可突破的最高等级。建筑等级越高，限制越宽松，允许英雄达到更高的境界。

这个上限值在 `AddHeroData()` 中计算，用于判定每个英雄的渡劫就绪状态。

## 2. 核心代码位置

| 文件 | 行号 | 说明 |
|------|------|------|
| `HeroBreakPageController.cs` | 133-167 | 渡劫台 — `AddHeroData()` 上限计算与状态判定 |
| `WhiteJadeBedPageController.cs` | 126-160 | 白玉床 — 完全相同的公式，仅建筑配置不同 |
| `AttrSystem_Util.cs` | 22-49 | `GetAttributePercent()` — 属性值归一化为百分比 |
| `CfgExt_BuildingCollection.cs` | 5-30 | `GetCfgByType()` — 按建筑类型获取配置（按Level排序取第一个） |

## 3. 计算公式

```csharp
var buildingCfg = baseBuildingPanel.GetBuildingConfig();
var baseVal = Cfg.Ins.property.P1306;
var increaseVal = AttrSystem.Ins.GetAttributePercent((uint)buildingCfg.AttributeList[0])
                   / 100 * buildingCfg.AttributeValueList[0];
var limit = baseVal + increaseVal;
```

### 3.1 变量拆解

```
limit = P1306 + (GetAttributePercent(AttributeList[0]) / 100 * AttributeValueList[0])
  │        │                │                                  │
  │        │                │                                  └── 建筑属性值系数（building表）
  │        │                └── 属性ID（building表 AttributeList 第一个元素）
  │        └── 全局基础等级上限（property表 P1306）
  └── 最终可升级的最大等级
```

### 3.2 逐步详解

**第一步：获取建筑配置**

`baseBuildingPanel.GetBuildingConfig()` 调用链：

```
BuildingBasePanelController.GetBuildingConfig()                     // BuildingBasePanelController_Util.cs:32
  → GetBuildingData().GetCfg()
  → building_collection.GetCfgByType(buildingType)                  // CfgExt_BuildingCollection.cs:9
    → 按 Type 过滤所有建筑配置 → 按 Level 升序排序 → 取第一个
```

返回的 `building` 配置对象中包含 `AttributeList` 和 `AttributeValueList` 两个数组，一一对应。**取 `[0]` 即该建筑关联的主属性。**

**第二步：读取基础值 P1306**

`Cfg.Ins.property.P1306` 是策划在 property 配置表中设定的**全局基础等级上限**，与建筑等级无关，是一个固定值。

**第三步：计算建筑加成值**

分三个子步骤：

> **3a. 取属性ID** → `buildingCfg.AttributeList[0]`

建筑配置中 AttributeList 数组的第一个元素。渡劫台和白玉床各自配置了不同的属性ID，用于关联各自的属性累积系统。

> **3b. 归一化为百分比** → `AttrSystem.Ins.GetAttributePercent(attrId)`

```csharp
// AttrSystem_Util.cs:22-49
long attrValue = attrMap[attrId];  // 从用户属性容器取出当前累积值

// 根据 attribute_bonus 表中该属性的 UnitServer 类型归一化：
switch (attrCfg.UnitServer)
{
    case 1: attrValue = attrValue * 100;   break;  // 固定值 → 百分比
    case 2: /* 不变 */                      break;  // 百分比 → 保持不变
    case 3: attrValue = attrValue / 10;    break;  // 千分比 → 百分比
    case 4: attrValue = attrValue / 100;   break;  // 万分比 → 百分比
}
return attrValue;
```

关键点：`attrMap[attrId]` 是玩家在游戏中通过**建筑升级、装备、其他系统**累积的属性总值。

> **3c. 转为等级加成** → `/ 100 * buildingCfg.AttributeValueList[0]`

将百分比值除以 100（还原为十进制倍数），再乘以建筑配置的 `AttributeValueList[0]`（该属性的等级加成系数）。

**第四步：合并得到 limit**

```
limit = baseVal + increaseVal
```

## 5. 状态判定优先级

在 `AddHeroData()` 中，状态判定有严格的优先级顺序：

```
优先级 1: HeroAction == HeroActionHeroLevelUpQueue  → InQueue  (排队等待)
优先级 2: HeroAction == HeroActionHeroLevelUpIng     → Ing      (渡劫中)
优先级 3: heroLevel < limit                          → Ready    (可渡劫)
优先级 4: heroLevel >= limit                         → Unlocked (需建筑升级)
```

`InQueue` 和 `Ing` 是**服务端状态**，优先生效。只有不在排队也不在渡劫中的英雄，才进入 limit 判定。

## 6. 建筑升级后的连锁反应

```
玩家升级渡劫台(E_116)
  → BuildingSystem 更新建筑数据
  → 建筑配置切换为下一等级（AttributeValueList[0] 变大）
  → 属性系统 attrMap 同步更新（累积值增加）
  → 发送 OnBuildingLevelUpSuccess 事件
  → HeroBreakPageController.RefreshView()
    → ResetHeroDatas() 重新遍历所有英雄
    → AddHeroData() 重新计算 limit
    → 原 Unlocked 英雄若 heroLevel < 新limit → 变为 Ready
```

## 7. 关键配置表依赖

| 配置表 | 字段 | 作用 |
|--------|------|------|
| `property` | `P1306` | 基础等级上限，硬编码的全局保底值 |
| `building` | `AttributeList[0]` | 建筑关联的属性ID，决定提升哪个属性来影响上限 |
| `building` | `AttributeValueList[0]` | 建筑属性值系数，随建筑等级递增 |
| `attribute_bonus` | `UnitServer` | 属性单位类型（固定值/百分比/千分比/万分比），决定归一化方式 |
| `hero_levelup` | `Rarity3/4/5HeroExp` | 各稀有度英雄渡劫所需的经验门槛 |

## 8. 渡劫台 vs 白玉床

| 维度 | 渡劫台 (E_116) | 白玉床 (E_114) |
|------|---------------|----------------|
| 控制器 | `HeroBreakPageController` | `WhiteJadeBedPageController` |
| 计算公式 | **完全相同** | **完全相同** |
| 建筑属性 | 各自的 `AttributeList[0]` | 各自的 `AttributeList[0]` |
| 行为动作 | `HeroActionHeroLevelUpIng/Queue` | `HeroActionWhiteJadeBedIng/Queue` |
| 事件 | `OnHeroLevelUpStart/Done` | `OnFastLevelUpStart/Done` |
| 升级方法 | `ForceHeroLevelUp()` | `ForceHeroFastLevelUp()` |
| 共享配置 | `P1306` | `P1306`（同一个基础值） |

两者共享同一套等级上限计算模型，但各自的建筑配置（`AttributeList[0]`、`AttributeValueList[0]`）不同，因此**建筑等级对上限的提升幅度可以独立调整**。

## 9. 示例推演

假设策划配置如下：

| 配置项 | 值 |
|--------|-----|
| `P1306` | `10`（基础可升到10级） |
| 渡劫台 Lv1 `AttributeList[0]` | `5001`（属性ID: 渡劫台等级加成） |
| 渡劫台 Lv1 `AttributeValueList[0]` | `5` |
| `attribute_bonus[5001].UnitServer` | `2`（百分比） |
| 玩家渡劫台1级的 `attrMap[5001]` | `0` |

```
baseVal = 10
increaseVal = GetAttributePercent(5001) / 100 * 5
            = 0 / 100 * 5
            = 0
limit = 10 + 0 = 10
→ 英雄可以升到10级
```

玩家升级渡劫台到 Lv2：

| 配置项 | 值 |
|--------|-----|
| 渡劫台 Lv2 `AttributeValueList[0]` | `10` |
| 玩家渡劫台2级的 `attrMap[5001]` | `100`（百分比=100%） |

```
baseVal = 10
increaseVal = GetAttributePercent(5001) / 100 * 10
            = 100 / 100 * 10
            = 10
limit = 10 + 10 = 20
→ 英雄可以升到20级
```

建筑等级越高 → `attrMap` 累积值越大 → `increaseVal` 越大 → `limit` 越大 → 更多英雄进入 Ready 状态。
