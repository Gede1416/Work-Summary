## 阶段一：AssetBundle 基础（预计 2 天）

### 学习目标
理解 AssetBundle 是什么、如何打包、如何加载、如何卸载。这是整个热更新系统的"物资单位"——所有热更文件本质上都是 AB 包。

### 理论学习（半天）
1. AssetBundle 概念：把 Unity 资源（Prefab/Texture/材质等）打包成 `.ab` 文件，运行时按需加载
2. 三种加载 API 的区别：`LoadFromFile` / `LoadFromMemory` / `LoadFromStream`（UnityWebRequest）
3. 依赖关系：包 A 依赖包 B，加载 A 前必须先加载 B
4. 卸载机制：`Unload(true)` vs `Unload(false)`，内存泄漏的常见原因
5. Manifest 文件：记录包名列表、依赖关系、Hash 值

### 动手实践（1 天）
1. 新建一个空白 Unity 项目
2. 创建几个 Prefab（Cube、Sphere），分别标记不同的 AssetBundle 名称
3. 写一个 Editor 脚本，调用 `BuildPipeline.BuildAssetBundles()` 打包
4. 写一个运行时脚本，用 `AssetBundle.LoadFromFileAsync()` 加载并实例化
5. 验证：先加载依赖包，再加载主包；尝试加载顺序错误观察报错

### 流程串联
回到架构文档 5.1 数据流图，只看 Build 端的 `AssetBundle 构建 → MD5 计算` 这一段，理解 AB 包打包后会产出哪些文件、为什么还需要 MD5。

## 阶段二：文件加载与寻址（预计 2 天）

### 学习目标
理解项目中文件如何被定位和读取——不是直接用 `AssetBundle.LoadFromFile`，而是通过 `FileLoader` 封装层，加上 md5 hash 后缀的文件命名规则。

### 理论学习
1. 项目中文件的实际命名：`{原名}_{md5}.{ext}`，同一个文件的不同版本通过 hash 区分
2. `FileLoader` 封装层：统一了 StreamingAssets 读取 + Persistent 目录读取 + 解密
3. `OriginFileToHashFileMgr`：把"原始路径"映射到"带 hash 的实际文件路径"
4. 加密文件的透明解密：调用层不感知文件是否加密，`FileLoader` 内部处理
5. 对照项目源码阅读：[资源加载结构.md](./资源加载结构.md) 中 FileLoader 部分

### 动手实践
1. 写一个 Editor 工具，对任意文件计算 MD5，并重命名为 `{name}_{md5}.{ext}`
2. 实现一个简单的 FileLoader：输入原始路径 → 查表得到 hash 文件名 → 读取文件
3. 加入简单的 XOR 加密：写文件时加密，读文件时解密，验证透明读写

### 流程串联
架构文档 2.3 节文件状态机，理解 `InStream`（Streaming 目录）和 `InPersistent`（Persistent 目录）两种文件位置的区别，以及 `FileLoader` 如何在这两处查找文件。

## 阶段三：HybridCLR 原理（预计 2 天）

### 学习目标
理解 IL2CPP 的局限（不能动态生成 IL 代码）、HybridCLR 如何突破这个限制、以及 DLL 分层（AOT/Hot）的编译模型。

### 理论学习
1. Unity 编译管线：C# → IL → IL2CPP → C++ → 原生代码
2. 为什么需要热更：IL2CPP 后不能 `Assembly.Load(byte[])` 动态加载普通 DLL
3. HybridCLR 的原理：补充 AOT 元数据 + 解释执行热更 DLL（而非 JIT）
4. `LoadMetadataForAOTAssembly`：为 AOT DLL 补充裁剪掉的元数据
5. AOT 层 vs Hot 层：哪些代码必须随包（AOT），哪些可以热更（Hot）
6. 对照项目源码：`AssemblyMgr` — `AddHotPatchAssembly` / `LoadAsync` / `LoadMetadataForAOTAssemblies`

### 动手实践
1. 安装 HybridCLR 到 Unity 项目（按官方文档走一遍）
2. 创建一个简单的 Hot 层脚本和一个 AOT 层脚本
3. 打包出 Hot DLL，运行时用 `Assembly.Load(byte[])` 加载
4. 在 Editor 模式和真机模式分别测试，理解 `#if !UNITY_EDITOR` 分支差异

### 流程串联
架构文档 4.2 节"AOT 到 Hot 层的完整调用链"，只看 `AssemblyMgr.LoadAsync()` 这一段，理解 DLL 字节码来自 Persistent 目录（由热更新下载的）。

## 阶段四：版本管理与 MD5 对比（预计 2 天）

### 学习目标
理解"哪些文件需要更新"是如何判断的——远端 md5.json 和本地 md5.json 的差异对比算法。

### 理论学习
1. 数据结构理解：`VersionDataForHotPatch`（远端）、`VersionDataForApp`（本地持久化）
2. `md5` vs `md5x` 的双重校验机制：
   - `md5`：持久化存储 hash（加密后计算），用于命名和比对
   - `md5x`：下载校验 hash（压缩+加密后计算），用于验证下载完整性
3. `CompareMD5` 三步算法：
   - a. `CheckFileRemoved`：本地有但远端已删除 → 清除
   - b. 逐文件对比 md5：本地不存在 / md5 不同 / 有未完成的下载 → 加入更新列表
   - c. 分流：`di=true` 或首次下载 → 立即下载；其他 → 延迟下载
4. 文件状态机：None → InStream → NeedDownload → Downloading → InPersistent

### 动手实践
1. 写一个简单的 `VersionDataForHotPatch` JSON 结构，手工构建测试数据
2. 实现 `CompareMD5` 的简化版：读取两个 JSON（本地+远端），输出差异列表
3. 验证 `CheckFileRemoved` 逻辑：远端删了文件，本地也要删
4. 实现 JSON 的序列化与反序列化

### 流程串联
架构文档 4.1 节热更包下载完整流程中 `CompareMD5` 所在的那一段（第 400-418 行的 rect 区域），理解这三步在整个流程中的位置。

## 阶段五：下载管线与文件处理（预计 3 天）

### 学习目标
理解文件从远端服务器到本地 Persistent 目录的完整链路，以及下载后的解密→解压→再加密处理。

### 理论学习
1. `UpdateController` 下载编排：`DownLoadMd5Info` → `CompareMD5` → `DoUpdateFileList`
2. `WWWLoader.CacheFile`：下载到临时目录 → MD5x 校验 → 成功后移动到 Persistent
3. `OnFileDownload_TryDecryptAndUnzipFile` 处理链：
   - 服务器上文件：原始 → 加密 → 压缩 → md5x
   - 下载后处理：解密 → 解压（如果 gs != 0）→ 再加密 → 持久化
   - 设计意图：本地存储始终是加密态，运行时透明解密
4. 并发控制：`UniTaskSemaphore` 限制同时下载数
5. 进度回调：`SetProgress(done, total, %)` 驱动 UI 进度条
6. 存盘节流：累计下载 150 个文件或 50MB 后才写一次 md5.json

### 动手实践
1. 用 Python/Node 搭一个简单的 HTTP 文件服务器
2. 实现文件下载：UnityWebRequest 下载单个文件
3. 加入 MD5 校验：下载完成后计算文件 MD5，与预期值比较
4. 实现加密/压缩/解密/解压处理链（用 Unity 自带的压缩库或第三方）
5. 实现简单的并发下载控制（信号量模式）
6. 处理下载失败重试

### 流程串联
架构文档 4.1 节完整时序图，跟踪一个文件从 `DoUpdateFileList` → `CacheFile` → `validator` → `SetNewFileInfo` → `OnFileDownload_TryDecryptAndUnzipFile` → `SaveVersionInfoToPersistentDirSync` 的完整路径。

## 阶段六：AOT/Hot 桥接与反射（预计 2 天）

### 学习目标
理解 AOT 层（`HotModule`）如何通过反射调用 Hot 层（`HotLifecycleMgr`），以及 Editor/真机两种模式的差异。

### 理论学习
1. `HotModule.OnEnterGameAsync()` 的完整流程：
   - `AssemblyMgr.LoadAsync()` 加载 DLL
   - `AssemblyMgr.GetHotPatchAssembly("HotLogic")` 获取 Assembly
   - `GetHotLogicFunc("OnHotDllEnterGameAsync")` 反射获取方法
   - `method.Invoke(null, parameters)` 跨层调用
2. Editor vs 真机的 Assembly 来源差异（`#if !UNITY_EDITOR`）：
   - Editor：`AppDomain.CurrentDomain.GetAssemblies()` 直接反射
   - 真机：`Assembly.Load(byte[])` 从字节码加载
3. 每帧 Update 的委托缓存：`logicUpdateDelegate` 避免每帧反射
4. `HotLifecycleMgr` 的启动流程：`ParseExcelAsync` → `ParseAudioCfgAsync` → `LogicApp.OnEnterGameAsync`

### 动手实践
1. 写一个简单的反射 Demo：动态加载 DLL → 反射获取类型 → 反射调用方法
2. 实现委托缓存：第一次反射获取方法，转为 `Action` 委托存起来，后续直接调用
3. 用 `UniTaskCompletionSource` 实现异步等待 Hot 层初始化完成
4. 对比项目中的 `HotModule.cs` 源码，逐行理解

### 流程串联
架构文档 4.2 节时序图，从 `GameInitObject` 一路走到 `LogicApp.ForEachSystemAsync(s.OnEnterGameAsync)`。

## 阶段七：Editor 构建管线（预计 2 天）

### 学习目标
理解打包时 Editor 端做了什么——HybridCLR 编译、DLL 复制、AssetBundle 构建、MD5 清单生成。

### 理论学习
1. `BuildHybridCLR.GenerateAll()`：调用 HybridCLR 的 `PrebuildCommand` 和 `CompileDllCommand`
2. `BuildHybridCLR.CopyDllByTarget()`：
   - AOT 元数据 DLL → `StreamingAssets/data/dll/`（带版本号 hash 命名）
   - Hot DLL → `Doc/dll/xxx.dll.bytes`（后续打入 AB 包）
3. `HotPatchBuildCmd`：调用 Unity 的 `BuildPipeline` 打包各平台 AB
4. `AbBuilderUtil_Md5`：计算所有输出文件的 MD5/MD5x，生成 `VersionDataForHotPatch` JSON
5. 构建产物：AB 包 + md5.json → 上传到远端服务器

### 动手实践
1. 写一个完整的 Editor 构建脚本：一键执行 HybridCLR 编译 → AB 打包 → MD5 生成
2. 验证 Hot DLL 是否正确生成（用 ILSpy/dnSpy 查看）
3. 验证元数据 DLL 是否正确裁剪并复制到 StreamingAssets
4. 手动将构建产物部署到本地 HTTP 服务器，验证下载流程能跑通

### 流程串联
架构文档 1.1 节分层架构图的 Editor 层（橙色区域），理解 `HotPatchBuildCmd`、`BuildHybridCLR`、`AbBuilderUtil` 三者的调用顺序和产物传递关系。
