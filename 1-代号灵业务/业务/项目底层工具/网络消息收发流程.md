# 网络消息收发流程

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                      业务层 (System)                         │
│  调用 Network.Ins.helper.XxxReqAsync(...) 发送请求           │
│  通过 RegisterMsgHandle 注册推送消息回调                     │
└────────────────────────┬────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────────┐
│                  自动生成层 (AutoGen)                         │
│  NetSenderHelper.cs  — 为每个 Req/Rsp 对生成强类型发送方法    │
│  CsProto.cs          — Protobuf 生成的 C# 消息类              │
└────────────────────────┬────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────────┐
│                   逻辑层 (Hot/Logic)                          │
│  Network.cs            — 网络主管理器，单例                   │
│  Network_BasicFunc.cs  — SendToServer() 发送入口              │
│  Network_BasicEvent.cs — OnReceive() 接收入口 + 分发          │
│  Network_BasicRegister.cs — RegisterMsgHandle / 错误处理       │
│  Network_Util.cs       — CreateMsg() / GetBodyName()          │
└────────────────────────┬────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────────┐
│                   传输层 (AOT/XKit)                           │
│  INetworkClient.cs     — 抽象基类，LZ4解压                    │
│  NetworkClientTcp.cs   — 基于 Telepathy 的 TCP 实现           │
│  NetworkWebsocket.cs   — WebSocket 实现                       │
│  NetworkUtil.cs        — LZ4压缩 / XOR混淆 / 校验和           │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 消息 ID 定义方式

消息 ID 直接使用 `_index.proto` 中 `CSMsgBody.body_oneof` 的 oneof 字段编号，不定义独立枚举：

```protobuf
// zb-common/protobuf/cs_proto/_index.proto

message CSMsgHead {
    int32  error_code = 1;     // Error = (表中的错误码 << 1) + 1
    int64  timestamp  = 2;
    uint32 sequence   = 3;     // 请求-回复 匹配用
    string msg        = 4;     // 服务端返回的可选消息
    int32  guide_progress = 5; // 引导步骤交互
}

message CSMsgBody {
    oneof body_oneof {
        // test
        CSTestReq mcs_test_req = 1;
        SCTestRsp msc_test_rsp = 2;

        // 内置
        CSPing mcs_ping = 10;
        SCPong msc_pong = 11;
        SCSyncTime msc_sync_time = 12;

        // login — 200-999
        CSFirstPkgReq  mcs_first_pkg_req  = 200;
        SCFirstPkgRsp  msc_first_pkg_rsp  = 201;
        // ...

        // game login — 1100-1499
        CSLoginGameServerReq mcs_login_game_server_req = 1120;
        // ...

        // init — 1500-1599
        SCInitBaseInfo msc_init_base_info = 1500;
        // ...

        // 业务模块 — 3000+
        // basic=3000, item=3300, hero=3600, building=3900,
        // equipment=3950, order=4000, cart=4100, quest=4200,
        // map=4500, mail=5100, chat=5500, activity=5600,
        // plant=5700, refining_furnace=6000
    }
}
```

**ID 分配规则**：每个业务模块预留约 300 个 ID 区间。新消息的 ID 在对应模块区间内取下一个未用的编号。

### 2.3 消息命名约定

| 类型 | 命名格式 | proto 字段名 | 说明 |
|------|----------|-------------|------|
| 客户端请求 | `CS{Action}Req` | `mcs_{action}_req` | 客户端发起，期望服务器回复 |
| 服务器回复 | `SC{Action}Rsp` | `msc_{action}_rsp` | 服务器对请求的回复 |
| 服务器推送 | `SCSync{Data}` | `msc_sync_{data}` | 服务器主动推送，无需请求 |
| 客户端单向 | `CS{Action}` | `mcs_{action}` | 客户端发送，不期望回复 |

### 2.4 顶层消息结构

```
CSMsg                           # 顶层封装
├── Head: CSMsgHead
│   ├── ErrorCode  (int32)      # 错误码
│   ├── Timestamp  (int64)      # 时间戳
│   ├── Sequence   (uint32)     # 序列号（请求-回复配对）
│   ├── Msg        (string)     # 服务端附加消息
│   └── GuideProgress (int32)   # 引导进度
└── Body: CSMsgBody
    └── body_oneof (oneof)      # 实际消息体（Req/Rsp/Sync 其一）
```

### 3.3 NetSenderHelper 模板输出示例

模板为每个 Req 消息生成两个版本（通过 `#if KITCORE_NETWORK_ONLY_CALLBACK` 切换）：

**版本一（仅回调）**：
```csharp
// 模板生成的代码 (ClientNetSender.cs)
public void TestReqAsync(CSTestReq sendBody,
        Action<SCTestRsp> onSuccess = null, Func<CSMsgHead, bool> onFail = null, int guideProgress = 0)
{
    CSMsg reqMsg = network.CreateMsg();
    reqMsg.Head.GuideProgress = guideProgress;
    reqMsg.Body.McsTestReq = sendBody;
    network.SendToServer(
        CSMsgBody.BodyOneofOneofCase.McsTestReq.ToString(), reqMsg,
        (rspMsg) => onSuccess?.Invoke(rspMsg.Body.MscTestRsp),
        (rspHead) => onFail != null ? onFail(rspHead) : false
    );
}
```

**版本二（UniTask 异步）**：
```csharp
// 返回 (CSMsgHead, SCTestRsp) 元组，可 await
public async UniTask<(CSMsgHead, SCTestRsp)> TestReqAsync(CSTestReq sendBody, ...)
{
    var tcs = new UniTaskCompletionSource();
    // ... 同上 SendToServer，回调中 tcs.TrySetResult()
    await tcs.Task;
    return (retRspHead, retRspBody);
}
```

**单向消息**（无 Req 后缀）：
```csharp
public void LogoutGameServer(CSLogoutGameServer sendBody, int guideProgress = 0)
{
    CSMsg reqMsg = network.CreateMsg();
    reqMsg.Head.GuideProgress = guideProgress;
    reqMsg.Body.McsLogoutGameServer = sendBody;
    network.SendToServer(
        CSMsgBody.BodyOneofOneofCase.McsLogoutGameServer.ToString(), reqMsg, null, null);
}
```

### 3.4 生成规则的判定

模板通过解析 `_index.proto` 中 `body_oneof` 的字段名称来判断消息类型：

- **字段名以 `mcs_xxx_req` 结尾** → 生成 `XxxReqAsync()` 方法（期望 `msc_xxx_rsp` 配对回复）
- **字段名以 `mcs_xxx` 结尾（无 req 后缀）** → 生成 `Xxx()` 方法（单向发送，不期望回复）
- **字段名以 `msc_xxx` 开头** → 不生成发送方法（这是服务器到客户端的消息）

## 四、消息发送流程

### 4.1 发送链路图

```
业务代码调用:
  Network.Ins.helper.TestReqAsync(sendBody, onSuccess, onFail)
          │
          ▼
  NetSenderHelper.TestReqAsync()          [AutoGen/ClientNetSender.cs]
    ├── network.CreateMsg()               → 创建 CSMsg，分配 Sequence
    ├── reqMsg.Body.McsTestReq = sendBody → 设置 oneof 消息体
    └── network.SendToServer(protoName, reqMsg, onSuccess, onFail)
          │
          ▼
  Network.SendToServer()                  [Network_BasicFunc.cs]
    ├── 检查网络状态 (status == Connected)
    ├── if protoName.EndsWith("Req"):
    │     RequestWaitingPanel.Show()       → 显示等待遮罩
    ├── if onSuccess != null:
    │     seqHandlerMap[seq] = SeqCallback → 注册序列号回调
    ├── reqMsg.WriteTo(MemoryStream)       → Protobuf 序列化
    └── networkClient.Send(buffer)         → 交给传输层
          │
          ▼
  INetworkClient.Send()                   [抽象方法]
          │
          ▼
  NetworkClientTcp.Send()                 [TCP 实现]
    └── NetworkUtil.ProcessSendData(data, length)
          │
          ▼
  NetworkUtil.ProcessSendData()           [AOT/NetworkUtil.cs]
    ├── if data.Length > 256:
    │     LZ4Codec.Encode()               → LZ4 压缩
    │     写入 4字节 guessUncompressedSize
    ├── else:
    │     直接拷贝（guessSize = 0，不压缩）
    ├── obfusSendData()                   → 从 offset 8 开始 XOR 106 混淆
    └── calcSum()                         → 计算 2字节 Internet 校验和，写入 offset 0
          │
          ▼
  最终数据包: [2B checksum][2B msgId][4B guessSize][...payload...]
          │
          ▼
  Telepathy 库 → TCP Socket 发送
```

### 4.2 关键代码路径

| 步骤 | 文件 | 方法 |
|------|------|------|
| 创建消息 | `Network_Util.cs:23` | `CreateMsg()` — 分配 Sequence，创建 CSMsgHead + CSMsgBody |
| 发送入口 | `Network_BasicFunc.cs:43` | `SendToServer()` — 注册回调、序列化、调用传输层 |
| 序列化 | `Network_BasicFunc.cs:103` | `reqMsg.WriteTo(MemoryStream)` — Google.Protobuf 序列化 |
| 传输层发送 | `NetworkClientTcp.cs` | `Send()` → `NetworkUtil.ProcessSendData()` |
| 压缩 | `NetworkUtil.cs:116` | `ZipUtil.Encode()` — LZ4 压缩（>256字节时启用）|
| 混淆 | `NetworkUtil.cs:71` | `obfusSendData()` — 从 byte[8] 起逐字节 XOR 106 |
| 校验 | `NetworkUtil.cs:29` | `calcSum()` — 16位 Internet Checksum |

### 4.3 Sequence 机制

- 每个 `CSMsg` 被创建时自动分配一个递增的 `Sequence`（`Network_Util.cs:10`）
- 计数器上限为 `1073741824`，到达后重置为 1
- 发送请求时，sequence 与回调函数绑定存入 `seqHandlerMap`
- 收到回复时，通过 sequence 匹配并调用对应的 `onSuccess` 回调

## 五、消息接收流程

### 5.1 接收链路图

```
  TCP Socket 收到数据
          │
          ▼
  NetworkClientTcp (Telepathy 收包)
          │
          ▼
  INetworkClient.__OnData(socketId, ArraySegment<byte> data)   [AOT]
    ├── 首包特殊处理：读取 4字节 连接结果码
    │     if retCode==0 → TriggerOnConnectTargetServerSuccess()
    │     else          → TriggerOnConnectTargetServerFail()
    ├── 后续包：
    │     ├── 读取 guessUncompressedSize (byte[2..5])
    │     ├── if guessedOutputLength != 0:
    │     │     LZ4Codec.Decode()          → LZ4 解压
    │     └── TriggerOnData(socketId, unzipData, length)
          │
          ▼
  Network.OnReceive(socketId, buffer, length)   [Network_BasicEvent.cs:54]
    ├── CSMsg.Parser.ParseFrom(buffer, 0, length)  → Protobuf 反序列化
    ├── 获取 reqSeqId = rspMsg.Head.Sequence
    ├── 从 seqHandlerMap 查找 seqCallbackInfo
    │
    ├── if ErrorCode != 0:  ────────────────── 错误处理 ──────────┐
    │     ├── GetErrorCodeInfo() → 解析 ErrorCode                   │
    │     ├── 查 Cfg.Ins.error_code 获取错误描述                    │
    │     ├── if seqCallbackInfo.onFail != null:                    │
    │     │     onFail(rspMsg.Head)   → 调用方自定义错误处理        │
    │     │     if return true → isErrorBeDone = true               │
    │     ├── if !isErrorBeDone:                                    │
    │     │     HandleErrorCode()  → 默认错误处理（弹窗/飘字）      │
    │     └── return                                                │
    │                                                               │
    ├── if Body == null → Error                                     │
    ├── GetBodyName(rspMsg) → 获取消息名称                          │
    ├── 同步服务器时间                                               │
    ├── if protoName.EndsWith("Rsp"):                               │
    │     RequestWaitingPanel.Hide()  → 关闭等待遮罩                │
    │                                                               │
    ├── if seqCallbackInfo.onSuccess != null:  ─── 回包匹配 ────────│
    │     onSuccess(rspMsg)  → 调用请求方的回调                     │
    │     return                                                    │
    │                                                               │
    └── msgHandlerFuncs[protoName]?.callback(rspMsg)  ─── 推送分发 ─│
         → 调用已注册的推送消息处理器                                │
```

### 5.2 消息分发的优先级

1. **错误码检查** — `ErrorCode != 0` 时优先处理错误
2. **序列号回包匹配** — 如果存在匹配的 `seqCallbackInfo.onSuccess`，调用后直接返回（不再走推送分发）
3. **推送消息分发** — 如果回包未被序列号回调消费，按消息名查找 `msgHandlerFuncs` 并调用

### 5.3 关键代码路径

| 步骤 | 文件 | 方法 |
|------|------|------|
| 原始数据接收 | `INetworkClient.cs:118` | `__OnData()` — LZ4 解压 |
| 数据触发 | `INetworkClient.cs:95` | `TriggerOnData()` — 事件回调 |
| Protobuf 解析 | `Network_BasicEvent.cs:63` | `CSMsg.Parser.ParseFrom(buffer, 0, length)` |
| 回包匹配 | `Network_BasicEvent.cs:65-71` | `seqHandlerMap[reqSeqId]` 查找 |
| 错误处理 | `Network_BasicEvent.cs:73-137` | ErrorCode 解析 → onFail → HandleErrorCode |
| 推送分发 | `Network_BasicEvent.cs:199-204` | `msgHandlerFuncs[protoName].callback(rspMsg)` |

## 六、新增消息的完整操作步骤

假设要在"道具模块"下新增一个 `ExchangeItem` 功能：

### Step 1：定义消息结构体

在 `zb-common/protobuf/cs_proto/game_server/item/item.proto` 中添加：

```protobuf
// 兑换道具请求
message CSExchangeItemReq {
    int32 item_id = 1;
    int32 count = 2;
}

// 兑换道具回复
message SCExchangeItemRsp {
    repeated ItemData items = 1;  // 兑换后获得的道具
}
```

服务器推送消息（如有需要）：

```protobuf
// 服务器主动推送道具变化
message SCSyncItemChange {
    repeated ItemData changed_items = 1;
}
```

### Step 2：注册消息 ID

在 `zb-common/protobuf/cs_proto/_index.proto` 的 `CSMsgBody.body_oneof` 中，在 item 模块的 ID 区间（3300+）内添加：

```protobuf
CSExchangeItemReq mcs_exchange_item_req = 3350;
SCExchangeItemRsp msc_exchange_item_rsp = 3351;
SCSyncItemChange  msc_sync_item_change  = 3352;
```

**注意**：
- `mcs_xxx_req` 和 `msc_xxx_rsp` 必须成对出现且 ID 连续（Req 在前）
- 确保 ID 不与同模块已有消息冲突
- 推送消息不需要与 Req 成对

### Step 3：运行代码生成

```bash
cd zb-common/protobuf
./build.bat
```

生成的产物：
- `dist/proto_generate/csharp/cs_proto/CsProto.cs` — 包含 `CSExchangeItemReq`、`SCExchangeItemRsp`、`SCSyncItemChange` 类
- `dist/proto_generate/csharp/ClientNetSender/ClientNetSender.cs` — 包含 `ExchangeItemReqAsync()` 方法

### Step 4：复制生成文件到 Unity

将生成的文件复制到客户端项目：

```bash
# CsProto.cs
cp zb-common/protobuf/dist/proto_generate/csharp/cs_proto/CsProto.cs \
   zb-client/client/Assets/Scripts/Hot/AutoGen/Gen/cs_proto/

# ClientNetSender.cs
cp zb-common/protobuf/dist/proto_generate/csharp/ClientNetSender/ClientNetSender.cs \
   zb-client/client/Assets/Scripts/Hot/AutoGen/Gen/ClientNetSender/
```

### Step 5：在业务系统中发送请求

在对应的 System 中调用自动生成的发送方法：

```csharp
// 例如 ItemSystem.cs
public async UniTask ExchangeItem(int itemId, int count)
{
    var sendBody = new CSExchangeItemReq
    {
        ItemId = itemId,
        Count = count
    };

    var (head, rsp) = await Network.Ins.helper.ExchangeItemReqAsync(sendBody);

    if (head.ErrorCode == 0)
    {
        // 处理成功：rsp.Items 包含兑换结果
        ApplyItemChanges(rsp.Items);
    }
}
```

### Step 6：注册推送消息处理器（仅推送消息需要）

如果有服务器推送消息（如 `SCSyncItemChange`），需在 `*_Net.cs` 中注册：

```csharp
// ItemSystem_Net.cs
public partial class ItemSystem : IBaseSystem
{
    private void DoRegisterNetworkEvents()
    {
        // 注册推送消息处理器
        Network.Ins.RegisterMsgHandle(
            nameof(CSMsgBody.BodyOneofOneofCase.MscSyncItemChange),
            this.OnSyncItemChange, this);
    }

    private void DoUnregisterNetworkEvents()
    {
        Network.Ins.UnRegisterMsgHandleByTarget(this);
    }

    private void OnSyncItemChange(CSMsg msg)
    {
        var proto = msg.Body.MscSyncItemChange;
        // 处理服务器推送的道具变化
        ApplyItemChanges(proto.ChangedItems);
    }
}
```

**注意**：请求/回复（Req/Rsp）消息**不需要手动注册处理器**——`NetSenderHelper` 生成的 `ReqAsync` 方法通过 `seqHandlerMap` 自动完成回包匹配。

## 七、关键类速查

### 7.1 逻辑层 (Hot/Logic)

| 类/文件 | 角色 |
|---------|------|
| `Network` (partial) | 网络主管理器，单例 `Network.Ins`。管理连接状态、消息分发、心跳 |

| `Network.RegisterErrorHandler()` | 注册特定错误码的处理器 |
| `Network.CreateMsg()` | 创建带 Sequence 的 CSMsg |
| `Network.GetBodyName()` | 从 CSMsg 提取消息名（如 `"MscSyncItemData"`） |
| `NetSenderHelper` | 自动生成类，为每个 Req 提供 `XxxReqAsync()` 方法 |

## 八、传输层数据格式

### 8.1 发送数据包结构

```
[byte 0-1]  2字节 校验和 (Internet Checksum)
[byte 2-3]  2字节 MsgId (废弃字段，实际不使用)
[byte 4-7]  4字节 guessUncompressedSize (0=未压缩)
[byte 8+]   payload (Protobuf 序列化数据，可能经 LZ4 压缩)
            从 byte 8 开始每字节 XOR 106 混淆
```

### 8.2 处理管道

```
发送: Protobuf序列化 → [>256B?]LZ4压缩 → XOR混淆(offset 8+) → 校验和(offset 0-1)
接收: 校验(忽略) → LZ4解压(根据guessSize) → Protobuf反序列化 → CSMsg对象
```

### 8.3 压缩阈值

- Payload ≤ 256 字节：**不压缩**，`guessUncompressedSize = 0`
- Payload > 256 字节：**LZ4 压缩**，`guessUncompressedSize = 原始大小`

### 8.4 首包特殊处理

TCP 连接建立后，第一个数据包为 4 字节连接结果码：
- `0` → 连接目标服务器成功
- 非 0 → 连接失败（错误码）

首包不经过 LZ4 解压，直接由 `INetworkClient.__OnData()` 处理。

## 九、错误处理机制

### 9.1 错误码编码

```
ErrorCode 原始值 = (配置表中的错误码 << 1) + 1   → Error
ErrorCode 原始值 = 配置表中的错误码 << 1          → Warning
```

判断方式：`errorCodeTemp % 2 == 1` 为 Error，`errorCodeTemp % 2 == 0` 为 Warning。

### 9.2 错误处理优先级

1. **发送方 onFail 回调** — `seqCallbackInfo.onFail(rspMsg.Head)`，返回 `true` 则跳过默认处理
2. **全局 ErrorHandler** — `errorHandlerMap[errorCode]`，由业务模块通过 `RegisterErrorHandler()` 注册
3. **默认处理** — Error 弹 MessageBox，Warning 飘字提示

### 9.3 常见错误码

| 错误码 | 含义 |
|--------|------|
| `AccountNotFound` | 账号不存在（Warning 级别，引导注册） |
| `VersionNotMatch` | 版本不匹配（Warning 级别，提示更新） |
